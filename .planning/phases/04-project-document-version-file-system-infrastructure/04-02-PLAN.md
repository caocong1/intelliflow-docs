  <action>
Create `packages/backend/src/modules/projects/` directory with two files:

**projects.service.ts** — Business logic functions:

1. `createProject(userId, body: {name, description?, department?})` — Insert project, auto-insert creator as owner in project_members (PROJ-08). Return project with memberCount=1.

2. `listProjects(userId, tab: "created"|"joined"|"all", page, pageSize, search?)` —
   - "created": projects where createdBy = userId AND isDeleted = false
   - "joined": projects where user is in project_members AND isDeleted = false (exclude ones user created)
   - "all": all non-deleted projects (visible to all authenticated users)
   - Include memberCount (count from project_members) and userRole (current user's role, null if not member)
   - Search: ilike on name
   - Return { data: ProjectListItem[], total: number }

3. `getProject(projectId)` — Get single project with memberCount. Return null if not found or deleted.

4. `updateProject(projectId, body: {name?, description?, department?})` — Update project fields, set updatedAt. Only owner should call (checked in route).

5. `deleteProject(projectId)` — Soft delete: set isDeleted=true, deletedAt=now. Only owner should call.

6. `listMembers(projectId)` — List all members with user displayName and username joined.

7. `addMember(projectId, userId, role?: ProjectRole)` — Insert into project_members. Default role "participant". Check user is not already a member.

8. `removeMember(projectId, userId)` — Delete from project_members. Cannot remove if user is the sole owner (check count of owners first).

9. `leaveProject(projectId, userId)` — Same as removeMember but self-initiated. If user is an owner, check there's at least one other owner remaining (PROJ-06).

10. `updateMemberRole(projectId, userId, role)` — Update member role. Must be called by an owner.

11. `isProjectOwner(projectId, userId)` — Helper: returns boolean.

12. `isProjectMember(projectId, userId)` — Helper: returns boolean.

Use db transactions where needed (e.g., deleteProject, leaveProject with owner check).

**projects.routes.ts** — Elysia routes with `requireAuth` (NOT requireAdmin — projects are user-facing):

```
GET    /projects                           — listProjects (tab, page, pageSize, search query params)
POST   /projects                           — createProject (body: name, description?, department?)
GET    /projects/:id                       — getProject (include membership check for access)
PATCH  /projects/:id                       — updateProject (owner only)
DELETE /projects/:id                       — deleteProject (owner only, soft delete)
GET    /projects/:id/members               — listMembers (member only)
POST   /projects/:id/members               — addMember (owner only, body: userId, role?)
DELETE /projects/:id/members/:userId       — removeMember (owner only)
POST   /projects/:id/leave                 — leaveProject (self)
PATCH  /projects/:id/members/:userId/role  — updateMemberRole (owner only, body: role)
```

For owner-only routes: check isProjectOwner inside the handler and return 403 if not.
For member-only routes: check isProjectMember and return 403 if not.

Use Elysia `t.Object` for body/query validation following existing patterns.

**packages/backend/src/index.ts** — Import and `.use(projectRoutes)` after workflowRoutes.
