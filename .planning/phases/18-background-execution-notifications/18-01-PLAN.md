  <files>packages/backend/src/modules/runtime/background.service.ts, packages/backend/src/modules/runtime/model-call.service.ts, packages/backend/src/modules/runtime/runtime.routes.ts, packages/backend/src/index.ts</files>

**A) Create `background.service.ts`** — the core orchestrator:

1. `executeDocumentPipeline(documentId: string, userId: string, startFromNodeId?: string)`:
   - Query nodeExecutions for this document, ordered by stepOrder
   - If `startFromNodeId` provided (retry case), skip nodes before it and skip already-completed nodes
   - Create a backgroundTask row (status: queued), then update to running with startedAt
   - Loop through nodes in order. For each node:
     - Skip if status is "completed" or "skipped"
     - Update node status to "in_progress" with startedAt
     - Execute based on nodeType:
       - `input_transform`: Skip — already confirmed by user before background execution starts
       - `desensitize`: Call existing desensitize service function (import from desensitize.service.ts)
       - `model_call`: Call new `executeModelCallBackground()` (see part B below)
       - `restore`: Call existing `executeRestore()` from restore.service.ts
       - `file_export`: Call existing export service function (import from export logic)
     - On success: mark node completed, update outputData, call advanceNode
     - On failure: mark node failed with errorMessage, mark backgroundTask failed, update document status to "failed", throw error to stop pipeline
   - On pipeline completion: mark backgroundTask completed with completedAt, update document status to "completed"
   - **Important**: Auto-select first model output when multiple models configured (set `selectedOutputKey` to the first key). Users can change selection later in workspace.

2. `detectOrphanTasks()`:
   - Query backgroundTasks where status = 'running'
   - For each orphan:
     - Update backgroundTask status to 'failed', errorMessage to '服务器重启，任务中断', updatedAt to now
     - Update any nodeExecutions for that document with status 'in_progress' to 'failed' with errorMessage '服务器重启，任务中断'
     - Update document status to 'failed'
     - (Notification creation will be handled by Plan 02 — for now just log a warning)

**B) Add background variant in `model-call.service.ts`**:

Add `executeModelCallBackground(documentId: string, nodeExecutionId: string, userId: string)`:
   - Reuse the existing model configuration loading logic from `executeModelCall`
   - Instead of returning an SSE ReadableStream, call the AI provider API directly and collect the full response
   - If multiple models are configured, execute them sequentially (or Promise.all for parallel), collect all results
   - Write results to nodeExecutions.outputData in the same format the SSE version uses on completion
   - Set selectedOutputKey to the first model's key (auto-select for background mode)
   - Mark node as completed with completedAt

**C) Add `POST /:documentId/start-background` endpoint in `runtime.routes.ts`**:
   - Verify user is project member (same pattern as existing routes)
   - Verify input_transform node is confirmed (status = completed)
   - Call `initDocumentExecution` if not already initialized
   - Update document status to 'in_progress'
   - Fire-and-forget: `executeDocumentPipeline(documentId, userId).catch(err => console.error(...))`
   - Return immediately: `{ status: "queued" }`

**D) Add startup hook in `index.ts`**:
   - Import `detectOrphanTasks` from background.service.ts
   - Call `await detectOrphanTasks()` before the server starts listening (or right after app setup, before `.listen()`)
   - Log the count of orphaned tasks found
