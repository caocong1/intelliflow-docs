**restore.service.ts:**

1. `executeRestore(documentId: string, nodeExecutionId: string, config: RestoreConfig)`:
   - Get input text from this node's inputData (the upstream model call output, resolved by runtime from selected output)
   - Load desensitize mappings from DB, filtering by the paired desensitize nodeExecution:
     - If config.pairedDesensitizeNodeId is set, find the nodeExecution for that workflow nodeId and query its mappings
     - Otherwise, load ALL desensitize mappings for this document
   - For each mapping, replace `placeholder` with `originalValue` in the text using string.replaceAll()
   - Track results per mapping: { placeholder, originalValue, sensitiveType, restored: boolean }
     - `restored: true` if placeholder was found and replaced
     - `restored: false` if placeholder was NOT found in text (may have been modified by AI)
   - Compute diff segments for highlighting:
     - Walk through restored text, mark ranges where replacement happened as "restored"
     - Mark ranges where placeholder still exists (failed) as "failed"
   - Store in outputData: { originalText (with placeholders), restoredText, restorations: Array<{placeholder, originalValue, sensitiveType, restored}> }
   - Return outputData

2. `updateRestoredText(documentId: string, nodeExecutionId: string, updatedText: string)`:
   - User manually corrected failed restorations via inline editing
   - Update nodeExecution.outputData.restoredText with the corrected text
   - Re-check all restorations to update restored status
   - Write final restored text to step directory
   - Return updated outputData

**restore.routes.ts:**
Routes under `/runtime/:documentId/restore/:nodeExecutionId`:
- `POST /execute` — calls executeRestore, returns { originalText, restoredText, restorations }
- `PUT /text` — body: { updatedText }, calls updateRestoredText (for manual corrections)

Register in runtimeRoutes.

**Two-phase flow:**

**Phase 1 — Execute Restore:**
- "开始恢复" (Start Restore) button -> calls POST /execute
- Show loading spinner during processing

**Phase 2 — Review Diff (per user decision: left/right split):**
- Left column: "脱敏版" (Desensitized Version) — original text with placeholders shown in amber/yellow highlights
- Right column: "恢复版" (Restored Version) — restored text with:
  - Successfully restored items: green background highlight (`bg-green-100 text-green-800`)
  - Failed items (placeholder still present): red background highlight (`bg-red-100 text-red-800`) + small edit icon
- Bottom summary bar: "已恢复 X 项, 失败 Y 项" (Restored X items, Failed Y items)
- Can reuse the diff display pattern from VersionDiff component but with inline spans instead of line-based diff

**Manual correction for failed items:**
- Click on a red-highlighted failed item in the right column -> transforms into an inline text input pre-filled with the placeholder text
- User types the correct value and presses Enter or clicks away
- On blur/enter: call PUT /text with the full updated text
- After correction, re-render: the item turns green if the placeholder is now gone
- "确认/下一步" only enabled when all failed items are resolved (restorations.every(r => r.restored))
  - OR show a warning dialog: "仍有 Y 项未恢复，确认继续？" (Y items still unrestored, confirm to continue?)

**Read-only mode:** Show the final restoredText with green highlights, no editing.
