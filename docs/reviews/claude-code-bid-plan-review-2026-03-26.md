## 一、总体判断

这个 plan 的方向是对的，尤其是下面三点判断基本成立：

1. 投标文件整体结构确实主要由招标文件和评分项决定，不存在真正通吃的固定模板。
2. 技术偏差表、技术应答表这类“条款逐条响应”内容最适合做标准化生成。
3. AI 的主要价值区确实集中在方案正文、服务方案、实施方案、培训方案、售后方案这些文字密集部分。

但当前版本还不能直接落地到 IntelliFlow。它对真实材料形态、证据附件、超大文档解析、表格/平台化输出、以及“无招标文件时如何工作”这几个关键问题处理得过于理想化。按现版本实施，前期能跑通演示，但在真实项目里会很快卡在输入不全、材料抽取不稳、证据链不闭环、导出不合规这四个点上。

## 三、主要问题与具体修改意见

### 1. 输入前提过窄，默认“招标文件必选”不符合真实场景

问题：
- 当前 plan 把“上传招标文件/技术规范书（必选）”作为流程入口。
- 但本次样本里，真正可作为招标依据的文件只有 `灾备中心基础设施租赁项目技术规范书-基础设施租赁部分-0210.docx` 一份。
- 其他大部分文件其实是历史投标文件、资格文件、售前方案、需求材料、PPT、CAD。

风险：
- 一旦客户手里只有招标公告、需求清单、历史案例，没有完整招标书，现流程就无法启动。
- 系统也无法把历史标书沉淀成可复用知识库。

修改建议：
- 将流程拆成两个入口模式：
  - `新项目生成模式`：有招标文件/技术规范书时使用，目标是生成当前项目的投标响应文件。
  - `样本沉淀模式`：只有历史标书、方案、证据材料时使用，目标是抽取结构、条款、表格、证据和模板，沉淀到资料库。
- 把“招标文件必选”改成“项目材料至少一种必选”，材料类型包括：
  - 招标文件/技术规范书
  - 历史投标文件
  - 客户需求整理
  - 售前方案
  - 产品资料
  - 证书/授权/审计/社保等证明材料
  - PPT、CAD、图片扫描件

### 2. 缺少“材料接入与归一化”阶段

问题：
- 当前 plan 从“输入转换”直接进入“信息脱敏”和“大纲生成”，跳过了最关键的文档解析层。
- 真实样本文档非常大，且包含大量图片、扫描页、内嵌对象、表格、页码引用。

证据：
- 北航商务技术文件：982 页，469MB，848 个图片对象，5 个嵌入对象。
- 灾备中心投标文件：926 页，205MB，527 个图片对象。
- 华科软投标文件：530 页，147.8MB，332 个图片对象。

风险：
- 直接把原始大文件送入模型，成本高、稳定性差、抽取不完整。
- 表格、扫描件、页码、附件目录容易丢失。

修改建议：
- 在现有“①输入转换”前新增 `阶段0：材料接入与归一化`。
- 该阶段至少输出以下产物：
  - `s0/material_manifest.json`
  - `s0/document_inventory.md`
  - `s0/parse_report.md`
  - `s0/ocr_queue.json`
  - `s0/table_index.json`
  - `s0/image_index.json`
- 该阶段需要完成：
  - 文档分类：招标、投标、资格、商务、技术、附件、方案、需求、产品资料、图纸
  - 页数/大小/图片数/嵌入对象统计
  - OCR 路由
  - 表格抽取
  - 附件目录识别
  - 文档切片与分卷

### 3. “程序性文件 = AI 无法生成”这个判断过于粗糙

问题：
- 当前 plan 把投标函、授权书、保证金、资质证书、财务报告、原厂授权等统一归入“AI 无法生成，需人工附件/模板填充”。

事实：
- 这些内容里确实不能靠 AI 编造，但很多内容可以被系统规则化装配，而不是完全手工。
- 北邮商务文件、北航资格文件里大量内容是“固定表单 + 企业主数据 + 证据附件”的组合。

修改建议：
- 把第一层内容重新拆成三类，而不是一刀切：
  - `规则填充型`：投标函、授权书、承诺函、资格审查表、中小企业声明、商务偏离表
  - `证据挂载型`：营业执照、资质证书、审计报告、纳税社保、检测证书、原厂授权
  - `人工确认型`：保证金信息、签章、日期、授权链、特殊声明
- 为这部分建立结构化主数据，而不是只依赖模板：
  - 公司主体信息
  - 法定代表人/授权人/项目经理信息
  - 财务年度数据
  - 资质证书清单与有效期
  - 历史业绩清单
  - 原厂授权与售后函模板

### 4. 缺少“条款-响应-证据”闭环模型

问题：
- 当前 plan 强调“技术偏差/应答表”可标准化，这点是对的，但没有把“证明材料绑定”设计成系统级数据结构。

事实：
- 北航商务技术文件中，“技术条款偏离表”后面紧跟“技术条款响应证明材料”。
- 北航资格文件中，很多资格条款后面直接对应检测证书、许可证、安全证书。
- 华科软投标文件中，自查表里已经出现“证明文件所在页码”。

风险：
- 如果只有文字回答，没有证据绑定，输出很容易“不像真实投标文件”，也无法支撑评审追溯。

修改建议：
- 在系统中引入统一对象：
  - `ClauseItem`：招标条款
  - `ResponseItem`：响应结论
  - `EvidenceItem`：证明材料
  - `EvidenceLink`：条款与证据的绑定关系
- 技术应答和资格应答都统一输出：
  - 条款编号
  - 条款原文
  - 是否满足
  - 偏离说明
  - 响应摘要
  - 证明材料名称
  - 证明材料页码/定位
  - 是否必须项

### 5. 输出不应只以 Markdown 为中心

问题：
- 当前 plan 的主要中间产物都是 `.md`。
- 但真实投标文件里大量核心内容其实是表格、平台表单、页码索引、附件清单。

证据：
- 北邮技术文件开头就是平台化价格表、货物部件表、服务费用表。
- 北邮商务文件、北航资格文件包含大量表单型内容。
- 华科软投标文件中明确写了“电子版采用 EXCEL 格式单独提供”。

修改建议：
- 生成结果采用“双轨输出”：
  - `叙述型内容`：Markdown / Docx 章节草稿
  - `结构化内容`：JSON / CSV / XLSX / 表单对象
- 建议至少增加这些中间产物：
  - `s3/chapter_plan.md`
  - `s4/clauses.json`
  - `s4/response_tables.xlsx`
  - `s4/evidence_matrix.csv`
  - `s5/chapters/*.md`
  - `s7/export_package_manifest.json`

### 6. “一个大模型节点一次性生成所有章节”风险过高

问题：
- 当前 plan 因为 IntelliFlow 节点数量固定，提出用一个“大模型节点”按大纲一次性生成所有章节。

风险：
- 上下文过长，易丢约束。
- 表格类内容和正文类内容混在一起，质量会同时下降。
- 后续修改某一章时无法局部重跑。

修改建议：
- 不要按“一个节点 = 一次性生成整份方案”设计。
- 改成“固定节点数，但节点内部分组执行”的方式：
  - `节点A：材料解析与条款抽取`
  - `节点B：规则填充与结构化表格生成`
  - `节点C：技术响应与证据绑定`
  - `节点D：方案正文分组生成`
  - `节点E：一致性校验与导出装配`
- 在 `节点D` 内按章节组生成，而不是整份一次出：
  - 供货与交付组
  - 实施与部署组
  - 运维与售后组
  - 培训与组织组
  - 安全与质量组

### 7. 质检环节设计偏轻，缺少合规校验项

问题：
- 当前质检只检查完整性、★条款覆盖和数据一致性。
- 真实投标项目需要更细的合规检查。

修改建议：
- 将 `⑥ 质检` 扩展为 `合规质检 + 交付质检` 两层。
- 必检项建议至少包括：
  - 招标评分项覆盖率
  - 必须项/★项/实质性条款覆盖率
  - 资格条款证据完备性
  - 证明材料页码是否存在
  - 产品型号、品牌、数量、服务年限是否跨章节一致
  - 金额、税率、单位是否一致
  - 是否出现未提供证据却写成“完全满足”
  - 是否引用了资料库里不存在的资质、业绩、人员、证书
  - 是否缺少独立文件卷宗，如资格文件、商务文件、技术文件

### 8. 项目分类方式需要调整，不能简单按“工程建设/设备采购/系统集成/简单采购”做价值判断

问题：
- 当前 plan 把项目 D 判断为“简单采购类，AI 生成价值低，基本是表格和承诺函”。

事实：
- `技术文件.docx` 实际有 246 页，包含完整技术方案、总体架构、设备部署、认证计费、漫游、安全可靠性、产品选型、售后承诺等。
- 这说明“政府采购 + 平台填报”并不等于“无技术方案”。

修改建议：
- 项目分类从“采购类型”改成“交付复杂度评分”。
- 维度建议改为：
  - 方案正文复杂度
  - 技术条款密度
  - 证明材料密度
  - 文件卷宗数
  - 平台表单占比
  - 产品选型复杂度
- 系统应基于复杂度选择生成策略，而不是基于项目标签做先验判断。

### 9. 缺少多源资料融合设计

问题：
- 当前 plan 主要考虑招标文件、公司信息、产品选型、历史标书。
- 但样本中还存在需求表、PPT、CAD。

修改建议：
- 明确支持以下补充资料并定义使用方式：
  - 需求整理表：用于提炼用户场景、数量边界、楼层/区域信息
  - 售前方案：用于复用章节结构、网络设计思路、拓扑与规划文本
  - PPT：用于提炼产品亮点、典型场景、产品清单
  - CAD：用于人工确认点位和区域规划，不建议直接自动生成正文，但应允许挂到“现场勘测/平面布置依据”中

### 10. 导出阶段过于抽象，缺少“卷宗装配”能力

问题：
- 当前 plan 的最后两步是“信息恢复”和“文件导出”，描述太轻。
- 真实投标交付经常不是一个文件，而是多个卷宗。

事实：
- 北航是资格文件 + 商务技术文件分卷。
- 北邮是商务文件 + 技术文件分卷。
- 灾备中心是单卷，但正文和附件层级非常深。

修改建议：
- 导出阶段明确支持：
  - 单卷模式
  - 资格 / 商务 / 技术多卷模式
  - 按招标方模板装配目录、封面、页眉页脚、页码、附件顺序
  - 导出 Word 包、PDF 包、平台导入表、附件清单
- 新增导出产物：
  - `s7/volume_manifest.json`
  - `s7/export_checklist.md`
  - `s7/attachment_order.csv`

## 四、建议替换后的流程

建议将原 plan 的“两阶段”扩展为“六阶段”：

### 阶段0：材料接入与归一化

输入：
- 招标文件、历史标书、需求文档、售前方案、产品资料、证据附件、PPT、CAD

输出：
- 材料清单
- 文档分类
- 切片结果
- OCR/表格/图片索引

### 阶段1：结构化抽取

任务：
- 抽取项目基本信息
- 抽取条款、评分项、必须项、表单项
- 抽取公司主数据、产品主数据、证据材料元数据

输出：
- `project_profile.json`
- `clause_catalog.json`
- `scoring_matrix.json`
- `evidence_catalog.json`

### 阶段2：投标蓝图规划

任务：
- 生成卷宗划分
- 生成章节大纲
- 标注每章的生成方式：规则填充 / 证据挂载 / LLM 生成 / 人工确认
- 标注缺失信息

输出：
- `bid_blueprint.md`
- `chapter_plan.json`
- `missing_info_checklist.md`

### 阶段3：规则填充与响应表生成

任务：
- 生成投标函、授权书、承诺函、资格表、报价表、偏离表、技术应答表
- 绑定页码和证明材料

输出：
- `form_fields.json`
- `response_tables.xlsx`
- `evidence_matrix.csv`

### 阶段4：方案正文分组生成

任务：
- 按章节组生成供货、实施、运维、售后、培训、安全、质量等正文
- 使用历史标书、售前方案和产品资料做检索增强

输出：
- `chapters/*.md`
- `chapter_review_notes.md`

### 阶段5：合规质检与卷宗导出

任务：
- 合规检查
- 数据一致性检查
- 证据完备性检查
- 按模板装配为单卷或多卷

输出：
- `qa_report.md`
- `volume_manifest.json`
- `export_package_manifest.json`

## 五、对原 plan 文本的直接修改建议

建议直接修改原文中的几个关键表述：

1. 把“上传：招标文件/技术规范书（必选）”改为“上传：项目材料（至少一种必选），优先为招标文件/技术规范书；若无，则进入样本沉淀模式”。
2. 在“① 输入转换”前新增“阶段0：材料接入与归一化”。
3. 把“第一层：程序性文件 → AI 无法生成，需人工附件/模板填充”改为“第一层：规则填充 + 证据挂载 + 人工确认混合处理”。
4. 把“输出：s4/技术应答书.md, s4/技术偏差表.md, s4/供货范围表.md”改为“输出：Markdown 草稿 + JSON/CSV/XLSX 结构化表格双轨结果”。
5. 把“一个大模型调用节点，让 AI 按大纲一次性生成所有章节”改为“固定节点数下按章节组生成，不做整份一次性生成”。
6. 把“简单采购类标书 AI 生成价值低”改成“需按复杂度评估生成价值，不按采购类型直接下结论”。
7. 把“文件导出格式不同，导出模板需要灵活配置”进一步展开为“支持多卷宗装配、平台表单导出、附件清单排序、Word 模板绑定和导出校验”。

## 六、结论

这份 plan 的抽象方向是对的，但更适合作为“第一版思路说明”，还不够支撑真实投标文件生产级落地。下一步不应直接实现原流程，而应先补齐三块基础能力：

1. 材料接入与结构化抽取。
2. 条款-响应-证据闭环。
3. 表格/卷宗/导出装配能力。

如果只补一个点，优先补“阶段0：材料归一化”。因为没有这个阶段，后面的大纲生成、技术响应、方案写作、质检、导出都会建立在不稳定输入上。
