**1. Create `undo-redo.ts`** — Snapshot-based undo/redo (Claude's discretion: snapshot approach over command pattern for simplicity):
- `createUndoRedo(initial, maxHistory=50)` returning `{ push, undo, redo, canUndo, canRedo }`
- See RESEARCH.md Pattern 9 for full implementation
- `push(snapshot)`: deep-clone current to undo stack, set new current, clear redo stack
- `undo()`: pop from undo stack, push current to redo stack, return previous snapshot
- `redo()`: pop from redo stack, push current to undo stack, return next snapshot
- Use `structuredClone()` for deep cloning

**2. Create `autosave.ts`** — Debounced save with queue (Claude's discretion: 1.5s debounce):
- `createAutosave(saveFn, delay=1500)` returning `{ trigger, status, lastSavedAt }`
- status signal: "idle" | "saving" | "saved" | "error"
- lastSavedAt signal: Date | null
- Queue pattern: if save in-flight and new data arrives, queue it. When in-flight completes, fire queued save with latest data.
- See RESEARCH.md Pattern 10 for implementation

**3. Wire into WorkflowEditor.tsx**:
- Initialize undo/redo with the loaded workflow snapshot
- After every state-changing operation (node add/delete, edge add/delete, config change, node drag END — not every pixel), call `undoRedo.push(store.getSnapshot())`

- After every state change, also call `autosave.trigger(store.getSnapshot())`
- The autosave `saveFn` converts the snapshot to WorkflowNodeDef[]/WorkflowEdgeDef[] and calls PUT /api/workflows/:id
