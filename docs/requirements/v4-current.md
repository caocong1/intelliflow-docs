# AI 文档生成平台 — 需求大纲（v4）

> 基于 2026-03-18 审批会议反馈，在 v3"基础节点自由编排"架构基础上，进一步明确底层执行机制和数据存储架构：
> 1. **混合存储架构**：数据库存储元数据（任务状态、脱敏映射、执行记录、文件索引等），文件系统存储内容文件（节点输出、用户上传、导出文件等）
> 2. **统一模型调用抽象**：模型调用抽象为统一接口，支持两种执行引擎——**CLI 命令行调用**（v1 首选）和 **API 直接调用**（v2 扩展），上层流程编排无需感知底层调用方式
> 3. **文件系统驱动**：节点间数据传递以服务器文件为介质，每次文档生成在指定目录下创建本次生成目录，所有节点的输入输出均为该目录下的文件
> 4. **脱敏规则提示词内嵌**：脱敏映射关系以变量替换规则的形式隐含在后续所有模型调用节点的提示词中，模型在处理时自动使用脱敏后的值

## 一、产品概述

### 1.1 产品定位

面向公司内部多部门使用的 AI 文档生成平台。用户以「项目」为单位组织工作，在项目内创建各类文档（招投标文档、解决方案、需求说明、技术方案等），通过 **5 种基础节点类型**灵活编排文档生成流程，管理员通过配置（提示词、模型选择、输入输出定义）即可定义任意 AI 处理行为，驱动多模型并行生成、对比、迭代，快速产出高质量文档。

### 1.2 核心价值

- **流程可编排**：管理员用 5 种基础节点（输入转换、信息脱敏、模型调用、信息恢复、文件导出）自由组合编排文档生成流程，同一节点类型可多次出现并配置不同行为，适配不同业务场景（招投标、技术方案、会议纪要等），无需开发介入
- **效率提升**：一次输入，多模型同时生成，省去逐个平台粘贴对比的重复劳动
- **质量保障**：通过配置模型调用节点实现质检、合规检查、一致性校验、查重检测等任意 AI 检查行为；多模型输出横向对比取各家之长
- **安全可控**：信息脱敏规则内嵌于提示词，敏感信息映射关系本地存储，确保真实敏感信息不上传给 AI
- **知识沉淀**：流程配置、生成历史全部留存，形成组织资产
- **协作共享**：项目成员共享文档与生成记录，避免信息孤岛

### 1.3 目标用户


| 角色    | 典型场景              |
| ----- | ----------------- |
| 售前/商务 | 招投标文档、解决方案、商务报价说明 |
| 产品经理  | PRD、竞品分析、用户调研报告   |
| 技术人员  | 技术方案、架构设计文档、接口说明  |
| 项目经理  | 项目计划、风险评估、会议纪要    |
| 管理层   | 战略规划、分析报告         |

## 二、功能模块

### 模块一：用户管理

#### 2.1.1 账号与认证（首选方案：企业微信）

系统作为企业微信内部 Web 应用接入，完全依赖企业微信完成用户认证和组织管理，不单独建设账号体系和组织架构管理页面：

- 用户从企业微信工作台点击应用入口，自动完成 OAuth 认证登录，无需手动输入账号密码
- 首次登录时自动创建系统账号，同步企业微信用户信息（姓名、头像、部门、职位等）
- 后续登录自动识别身份，静默完成认证
- 企业微信外部浏览器访问时，支持企业微信扫码登录
- 用户信息和部门数据以企业微信通讯录 API 为主数据源，不提供独立组织管理页面，但系统保留必要的用户/部门缓存快照用于审计和历史追溯
- 需要查询用户或部门时（如项目邀请成员），直接调用企业微信接口检索
- 离职人员的历史操作记录保留，展示"已离职"标记

> **备选方案**：如未来需要脱离企业微信独立部署，可扩展为独立账号体系（邮箱/手机号注册、账号密码登录、SSO 单点登录）和独立的组织部门管理模块。

- **项目负责人退出限制**：项目负责人不可直接退出项目，必须先将负责人角色移交给其他成员后才能退出；如果是唯一的项目负责人，必须先指定新的负责人
- 项目内角色：


| 项目角色  | 权限                                                     |
| ----- | ------------------------------------------------------ |

| 项目参与者 | 创建文档、编辑/生成/查看**对自己可见的文档**（见 2.7.6 文档可见性），资料库的查看/上传/编辑  |


- 项目创建人默认为项目负责人
- 可添加多个项目负责人
- 项目负责人可将自己的负责人角色转让给其他项目成员
- 查看项目成员列表及其角色

#### 2.2.3 项目主页

根据用户角色展示不同视图：

- **项目负责人视图**：顶部菜单包含「文档列表」「进行中任务」「成员管理」「资料库」「项目设置」，默认展示文档列表
- **项目参与者视图**：顶部菜单包含「文档列表」「进行中任务」「资料库」，默认展示文档列表
- **进行中任务视图**：展示当前用户在本项目中正在执行流程但尚未完成的文档，按最后操作时间排序

> 人工审核相关的待办功能（待审核、被驳回）将在后续版本支持。

#### 2.2.4 项目资料库

项目内置的资料管理中心，用于存放与本项目相关的参考资料（如招标文件、旧版方案、行业报告等）。流程中通过输入转换节点或模型调用节点的输入配置引用资料库中的资料。

- **资料上传**：支持上传 PDF / Word / Excel / 文本等常见格式
- **自动处理**：
  - 文件上传后自动进行内容解析（提取文本内容）
  - 自动进行 RAG 处理（文本切片、向量化索引），使 AI 生成时可精准检索相关段落
- **资料分类管理**：
  - 支持对资料进行分类管理（如：招标文件、历史方案、行业报告、公司资质等）
  - 管理员可自定义分类标签
  - 资料列表支持按分类筛选查看
- **资料管理**：
  - 资料列表查看（文件名、大小、分类、版本号、上传人、上传时间、处理状态）
  - 处理状态：上传中 / 解析中 / 就绪 / 解析失败
  - 资料预览（查看解析后的文本内容）
  - 删除资料（进入回收站，可恢复）
  - 资料支持版本更新（上传新版本，旧版本保留为历史版本）
- **回收站**：删除的资料进入回收站，项目负责人可在回收站中恢复或彻底删除
- **资料复用**：同一项目内的所有文档均可引用资料库中的资料，避免重复上传

- 文档类型排序（拖拽排序或权重排序）

#### 2.3.2 文档类型定义

每个文档类型包含以下配置项：


| 配置项  | 说明              | 示例                            |
| ---- | --------------- | ----------------------------- |
| 类型名称 | 文档类型的显示名称       | 「招投标技术方案」                     |

#### 2.3.3 文档类型与流程的关系

- **流程与文档类型强绑定**：创建流程时必须选择所属文档类型，流程归属于特定文档类型
- 一个文档类型可拥有多个流程（1:N），各流程的差异体现在：包含哪些节点、节点出现次数和顺序、以及同一类型节点上附带的不同配置（提示词模板、变量、模型选择等）
- 每个文档类型可设置一个**默认流程**（新建文档时自动推荐）
- 用户新建文档时，选择文档类型后只看到该类型下的可用流程：
  1. 该类型的默认流程（推荐标记）
  2. 该类型下其他可用流程

### 模块四：流程编排配置

> 核心模块。管理员在此为特定文档类型创建流程——从 5 种基础节点类型中选择、添加、排序、配置，编排成适合特定业务场景的工作流。系统不内置流程模板，管理员从空白开始自由编排。

#### 2.4.1 底层执行机制

> **v4 新增**。本节描述流程执行的底层存储架构、模型调用方式和脱敏机制，是理解后续节点行为的基础。

##### 数据存储架构：数据库 + 文件系统

系统采用**混合存储架构**，数据库管理元数据，文件系统存储内容：

| 存储位置 | 存储内容 | 设计考量 |
| ------- | ------- | ------- |
| **数据库** | 生成任务记录（任务ID、状态、创建人、时间等）；节点执行状态与流转记录；脱敏映射关系（加密存储）；文件索引（路径、名称、类型、大小、所属节点）；模型调用日志（调用方式、耗时、Token消耗、状态）；版本记录与快照索引 | 结构化查询、状态管理、事务一致性、权限控制、审计统计 |
| **文件系统** | 节点输出的实际内容文件（.md/.json等）；用户上传的原始文件；导出的最终文件（.docx/.pdf等）；命令行执行的 stdout/stderr 日志文件 | 大文本不适合存DB；CLI agent 天然读写文件；便于模型直接操作 |

**数据库与文件系统的协作关系**：
- 文件系统中的每个文件在数据库中有对应的**文件索引记录**（文件路径、所属任务、所属节点、文件类型、创建时间等）
- 节点执行状态、流转逻辑、权限判断等通过数据库查询，不依赖文件系统状态
- 脱敏映射关系存储在数据库中（加密字段），比文件存储更安全，支持权限控制和审计追踪
- 节点执行时，系统从数据库获取任务上下文和文件索引，从文件系统读取实际内容

##### 文件系统驱动的数据流

每次用户创建文档并启动流程时，系统在服务器**指定根目录**下创建本次生成的**工作目录**，目录名称由系统按规则生成（如 `{项目编码}/{文档ID}/{时间戳}/`）。流程中所有节点的输入和输出均以该工作目录下的文件为介质：

```
/data/generations/{项目编码}/{文档ID}/{时间戳}/
├── s1/                          ← 输入转换节点输出
│   └── 需求清单.md
├── s2/                          ← 模型调用节点输出（可并行多模型）
│   ├── 需求报告-glm5.md
│   ├── 需求报告-kimi-k2.5.md
│   ├── 需求报告-MiniMax-M2.5.md
│   └── 需求报告-deepseek-v3.2.md
├── s3/                          ← 交叉对比节点输出
│   └── 交叉对比-需求分析报告.md
├── desensitize/                 ← 脱敏映射本地副本（主存储在数据库，不传给AI）
│   └── mapping.json
└── export/                      ← 最终导出文件
    └── 最终报告.docx
```

- 每个节点（步骤）对应一个子目录（如 `s1/`、`s2/`、`s3/`……），节点的输出文件写入对应子目录
- 下游节点通过读取上游子目录中的文件获取输入数据
- 节点的输出文件名在节点配置中预先定义（如输出文件名为 `需求报告-glm5.md`）
- 多模型并行调用时，各模型输出到同一子目录下的不同文件
- 文件系统中的每个文件在数据库中有对应的文件索引记录，便于查询和管理

##### 模型调用引擎：统一抽象，双引擎实现

模型调用节点通过**统一的调用抽象层**执行 AI 模型。上层流程编排只关心"输入→提示词→输出"，不感知底层调用方式。系统支持两种执行引擎，通过模型配置中的 `invoke_type` 字段切换：

| 执行引擎 | invoke_type | 版本 | 说明 |
| ------- | ----------- | --- | --- |
| **CLI 命令行调用** | `cli` | v1（首选） | 通过命令行 agent 方式调用（如 `claude -p "提示词"`），提示词中指定输入文件路径和输出文件路径，agent 直接读写服务器文件 |
| **API 直接调用** | `api` | v2（扩展） | 通过 HTTP API / SDK 直接调用模型接口，系统负责读取输入文件内容组装请求、接收响应后写入输出文件 |

**v1 CLI 命令行调用（首选方案）**

系统将节点配置的提示词组装后，以命令行 agent 方式执行：

```bash
claude -p "你是一个需求分析专家。请阅读 /data/generations/.../s1/需求清单.md 的内容，
按照以下要求生成需求分析报告，并将结果输出到 /data/generations/.../s2/需求报告-glm5.md。
...（具体要求）..."
```

- 提示词中**明确指定输入文件路径**（读取哪些上游节点的输出文件）和**输出目录/文件路径**（写入到当前节点的输出目录）
- 系统负责将节点配置中的变量（如 `{{输入目录}}`、`{{输出目录}}`、`{{输出文件名}}`）替换为实际的文件系统路径
- 命令行调用支持不同模型的适配（通过模型配置中的命令行模板定义）
- 调用过程的 stdout/stderr 日志记录到文件系统，执行状态和耗时等元数据写入数据库

**v2 API 直接调用（后续扩展）**

```
系统读取输入文件内容 → 组装 API 请求（提示词 + 文件内容）→ 调用模型 API
→ 接收流式响应 → 写入输出文件 → 更新数据库记录
```

- 系统自动读取输入文件内容，将内容嵌入提示词后通过 API 发送
- 模型响应内容由系统写入对应的输出文件，上层流程无感知差异
- 适用于不支持命令行调用或需要更精细控制（如流式 SSE、Token 精确统计）的场景
- API 调用配置复用现有 Provider 管理（API 地址、认证信息、模型参数等）

**统一的调用结果处理**：

无论使用哪种引擎，调用完成后系统统一执行：
1. 检查输出文件是否成功生成
2. 在数据库中记录：调用方式（cli/api）、使用模型、执行耗时、Token 消耗、成功/失败状态
3. 更新文件索引记录（文件路径、大小、类型）
4. 更新节点执行状态，触发下游流转

##### 脱敏规则的提示词内嵌机制

> **v4 关键变更**：信息脱敏不再作为独立的内容处理步骤（先替换内容再传给AI），而是将脱敏映射规则以变量替换说明的形式**嵌入到后续所有模型调用节点的提示词中**，由 AI 模型在处理时自动使用脱敏值。

工作机制：

1. **脱敏节点**：用户（或本地AI辅助）定义敏感信息映射表，例如：
   ```
   北京绿色苹果技术有限公司 → <!-- 000001 -->
   张三 → <!-- 000002 -->
   1500万元 → <!-- 000003 -->
   ```
   映射表**加密存储在数据库中**（同时在工作目录 `desensitize/mapping.json` 保留本地副本供恢复节点使用），**不上传给线上AI**

2. **后续模型调用节点的提示词自动注入脱敏规则**：系统在组装提示词时，自动将脱敏映射以规则说明的形式附加到提示词中，例如：
   ```
   【脱敏规则】以下信息已脱敏处理，在你的输出中请使用脱敏后的代号，不要尝试还原：
   - <!-- 000001 --> 代表某公司名称
   - <!-- 000002 --> 代表某人名
   - <!-- 000003 --> 代表某金额
   注意：<!-- --> 标记代表脱敏信息，不作为深度思考的依据，直接保留原样输出即可。
   ```

3. **信息恢复节点**：纯本地处理，读取 `desensitize/mapping.json`，将输出文件中的 `<!-- 000001 -->` 等占位符替换回真实值

此机制的优势：
- 脱敏规则随提示词传递，无需预处理原文内容，输入文件保持原始可读性
- AI 模型在生成过程中即使用脱敏代号，无需二次替换
- 脱敏规则描述中只提供类型说明（"某公司名称"），不泄露真实值

#### 2.4.2 流程编排示例

> 以下示例仅供参考，帮助理解如何用基础节点组合出不同场景的流程。实际流程由管理员从基础节点自由编排，不受示例限制。

> **脱敏分层原则**：信息脱敏在所有 AI 处理之前定义映射规则，该规则自动注入到后续所有模型调用节点的提示词中；信息恢复在所有 AI 处理之后，将占位符替换回真实值。

  │ → 模型调用[MiniMax-M2.5]:读取 s1/ → 输出 s2/需求报告-MiniMax-M2.5.md│
  │ → 模型调用[deepseek-v3.2]:读取 s1/ → 输出 s2/需求报告-deepseek-v3.2.md│
  │                                                                      │

  → 信息恢复（读取 mapping.json，替换占位符为真实值）

#### 2.4.3 流程管理

- **创建流程**：选择所属文档类型（必选）→ 填写流程名称和描述 → 进入流程编辑器，从空白开始编排节点
- 编辑流程
- 启用 / 停用流程（停用后不可用于新建文档，已在使用的文档不受影响）
- 删除流程（仅无关联文档时可删除）
- 流程列表与搜索（支持按所属文档类型筛选）
- 流程复制（基于已有流程快速创建变体，复制后可修改所属文档类型）
- 一个文档类型下可创建多个流程（1:N）

#### 2.4.4 流程编辑（系统管理员）

系统管理员在流程编辑器中从空白开始编排流程：

- 从 5 种基础节点类型中选择添加，**同一类型可多次添加**（如一个流程中可包含多个模型调用节点，各自配置不同行为）
- 编排节点顺序和连接关系
- **配置每个节点的具体参数**（详见 2.4.5 各节点类型说明）：
  - 输入转换节点：配置输入表单字段、文件上传选项、输出目录和文件名定义
  - 信息脱敏节点：配置脱敏规则类型、脱敏占位符格式
  - 模型调用节点：配置节点显示名称、输入文件路径（上游哪些目录/文件）、提示词模板、模型选择、输出目录和文件名、命令行调用模板
  - 信息恢复节点：配置恢复前后对比展示方式
  - 文件导出节点：配置导出格式模板文件、内容映射规则
- **节点输出文件配置**：每个节点需定义输出的文件名和文件类型（如 `需求清单.md`、`需求报告-glm5.md`），系统自动将文件写入该节点对应的步骤子目录
- 变量系统：提示词中可使用 `{{变量名}}` 插值，系统内置变量包括：
  - `{{工作目录}}` — 本次生成的根目录路径
  - `{{输入目录}}` — 当前节点配置的输入文件所在目录路径
  - `{{输出目录}}` — 当前节点的输出子目录路径
  - `{{输出文件名}}` — 当前节点配置的输出文件名
  - `{{脱敏规则}}` — 系统自动从 `desensitize/mapping.json` 生成的脱敏说明文本（仅在脱敏段内的节点中可用）
  - 其他由上游节点输出定义的业务变量
- 流程校验（系统自动检查）：
  - 流程必须以「输入转换」节点开始
  - 流程必须以「文件导出」节点结束
  - 流程必须包含至少一个「模型调用」节点
  - 「信息脱敏」与「信息恢复」必须成对出现
  - 「信息恢复」前必须有「信息脱敏」
  - 每个节点必须定义输出文件名，且同一步骤目录下文件名不可重复
- 流程预览：可视化展示节点流转图，展示文件流向

#### 2.4.5 基础节点类型（5 种）

每个节点对应用户操作的一个**页面步骤**。用户在页面上执行操作、点击"确认/下一步"后流转到下一个节点。每个节点执行完成后，输出文件写入该节点对应的步骤子目录。

##### ① 输入转换

用户在此页面填写需求、上传文件，系统将输入内容转换为结构化文件存储到工作目录。

- 页面展示可配置的输入区域组合：文字输入框、结构化表单字段（公司名/项目编号/日期等）、文件上传区域
- 上传文件支持 Word / PDF / 图片 / 音频 / 视频，系统自动解析为文本（PDF/Word 提取文字，图片 OCR，音视频语音转文字）
- 用户可查看文件解析结果并修改
- 确认后：输入数据按配置定义的格式写入步骤子目录（如 `s1/需求清单.md`），文件格式和命名由节点配置定义

**管理员可配置项：**

| 配置项 | 说明 |
| ---- | ---- |
| 输入表单字段 | 字段名、类型（文本/数字/日期/下拉选择等）、是否必填、默认值 |
| 文件上传选项 | 支持的文件类型、数量限制、解析方式 |
| 输出文件定义 | 输出的文件名（如 `需求清单.md`）、文件类型、内容模板格式 |

##### ② 信息脱敏

定义敏感信息的脱敏映射规则，以占位符形式标记，映射关系本地存储。该映射规则会被系统自动注入到后续所有模型调用节点的提示词中。

- 使用**本地私有部署的 AI 模型**辅助识别文档中的敏感信息（公司名/人名/金额/联系方式/地址等）并高亮标记
- 用户逐条确认是否需要脱敏，设置脱敏占位符（如 `<!-- 000001 -->`、`<!-- 000002 -->` 等系统自动编号）
- 可手动标记 AI 未识别的敏感信息
- **原始信息↔占位符的映射关系加密存储在数据库中**（工作目录 `desensitize/mapping.json` 保留本地副本供信息恢复节点使用），不随内容上传给线上 AI
- 映射规则以脱敏说明文本的形式（仅包含占位符和类型描述，不包含真实值）自动注入到后续模型调用节点的提示词中
- 多轮回退后映射关系保持一致（映射表与脱敏节点执行版本绑定）
- 确认后：脱敏映射关系确定，后续所有模型调用节点的提示词将自动包含脱敏规则说明

**管理员可配置项：**

| 配置项 | 说明 |
| ---- | ---- |
| 脱敏规则 | 识别的敏感信息类型（公司名/人名/金额/联系方式/地址等） |
| 占位符格式 | 占位符的格式规则（默认 `<!-- 编号 -->`） |
| 脱敏说明模板 | 注入到后续提示词中的脱敏规则说明文本模板 |
| 本地模型选择 | 仅可选择本地私有部署的模型（用于辅助识别敏感信息） |

**脱敏提示词注入示例：**

当脱敏节点定义了以下映射：
```json
{
  "<!-- 000001 -->": { "type": "公司名称", "original": "北京绿色苹果技术有限公司" },
  "<!-- 000002 -->": { "type": "人名", "original": "张三" },
  "<!-- 000003 -->": { "type": "金额", "original": "1500万元" }
}
```

系统在后续模型调用节点的提示词中自动注入（`original` 字段不会出现在注入内容中）：
```
【脱敏规则】以下标记代表脱敏信息，在你的输出中请直接保留这些标记原样输出，不要尝试推测或还原其真实内容：
- <!-- 000001 --> 代表某公司名称
- <!-- 000002 --> 代表某人名
- <!-- 000003 --> 代表某金额
注意：<!-- --> 格式的标记不作为深度思考的依据，遇到时直接原样保留即可。
```

##### ③ 模型调用

**核心节点**——通用的 AI 处理节点。通过统一调用抽象层执行模型（v1 CLI 命令行 / v2 API 直接调用），提示词中指定读取的输入文件路径和输出文件路径。一个流程中可包含多个模型调用节点，各自通过不同的配置承担不同任务。

- 页面分为三个区域：
  - **步骤说明区**：展示本步骤的任务描述（来自节点配置的显示名称和步骤说明，如"根据招标要求生成投标响应"）+ 来自上游节点的输入文件列表摘要，用户可展开查看完整文件内容
  - **变量填写区**：节点配置中定义的变量，部分由上游数据自动填充，未填充的变量以表单形式展示让用户补充（如无需填写则此区域不显示）
  - **生成区**：模型选择面板 + 生成结果展示区
- 提示词对用户**默认隐藏**（已由节点配置配置好），高级用户可点击「查看/编辑提示词」展开
- **底层执行**：系统将提示词模板中的变量替换为实际值（包括文件路径、脱敏规则等），通过统一调用抽象层执行：
  - **v1 CLI 方式**：`claude -p "{{组装后的完整提示词}}"`，agent 直接读写服务器文件
  - **v2 API 方式**：系统读取输入文件内容嵌入请求体，调用模型 API，将响应写入输出文件
  - 提示词中包含：读取哪些输入文件、执行什么任务、输出到哪个文件路径、脱敏规则说明（如在脱敏段内）
  - 调用完成后统一更新数据库中的执行记录和文件索引
- **模型选择**：默认「系统推荐」1 个最优模型，用户可展开切换为「多模型对比」模式手动多选。模型展示增加标签（速度优先 / 质量优先 / 低成本），并显示预估生成耗时
- **多模型并行调用**：选择多个模型时，系统并行发起多个命令行调用，各模型输出到同一步骤目录下的不同文件（如 `s2/需求报告-glm5.md`、`s2/需求报告-kimi-k2.5.md`）
- 各模型通过流式输出展示，每个模型有独立输出卡片，状态依次为：等待中 → 思考中(动画) → 生成中(内容流式输出) → 已完成/失败
- 支持切换查看不同模型输出（Markdown 渲染视图 / 源码视图）
- 单个模型失败可单独重试，已成功模型的输出保留不受影响
- 多模型时支持左右分栏对比
- **引用来源展示**：生成结果中标注引用来源（来自哪份参考资料的哪个段落），用户可查看每段内容的引用依据；导出时可选附带参考来源附录；用户可剔除某份资料后重新生成
- 用户选定最佳输出作为本节点最终输出
- 确认后：选定的输出文件作为本节点最终输出，流转给下游节点，所有模型输出文件保留

**管理员可配置项：**

| 配置项 | 说明 |
| ---- | ---- |
| 节点显示名称 | 面向用户的步骤名称（如"AI生成""合规检查""提取要求"等） |
| 节点描述/步骤说明 | 面向用户的任务描述文字 |
| 输入文件配置 | 指定读取的上游步骤目录和文件（如 `s1/需求清单.md`、`s2/*.md`），支持通配符 |
| 输出文件配置 | 输出的步骤目录编号、文件名模板（如 `需求报告-{模型名}.md`）、文件类型 |
| 提示词模板 | System Prompt + User Prompt，支持 `{{变量名}}` 插值，包括 `{{输入目录}}`、`{{输出目录}}`、`{{脱敏规则}}` 等系统变量 |
| 调用方式配置 | v1 CLI：命令行执行模板（如 `claude -p "{{提示词}}"`）；v2 API：复用 Provider 配置。不同模型可有不同的调用方式 |
| 变量定义 | 变量列表（变量名、类型、来源：上游文件自动提取 / 用户手动输入） |
| 模型选择 | 可用模型范围、默认推荐模型、是否允许用户切换模型、是否允许多模型对比模式 |
| 输出格式 | 输出格式（自由文本 / 结构化 JSON / 检查报告等） |
| 交互模式 | 是否需要用户确认后才流转（默认是）、是否展示流式输出、用户可否编辑提示词 |

> **配置示例**：同一个模型调用节点类型，通过不同配置可实现完全不同的功能：
>
> | 配置实例 | 显示名称 | 输入文件 | 输出文件 | 提示词要点 |
> | ------ | ------ | ------- | ------- | ------- |
> | AI 生成 | "生成投标响应" | `s1/需求清单.md` | `s2/需求报告-{模型名}.md` | 根据要求和参考资料生成文档内容 |
> | 交叉对比 | "多模型对比分析" | `s2/*.md` | `s3/交叉对比-需求分析报告.md` | 对比多个模型输出，取长补短合成最优版本 |
> | AI 质检 | "质量检查" | `s3/交叉对比报告.md` | `s4/质检报告.md` | 按审核标准逐条检查，输出各维度得分和问题清单 |
> | 合规检查 | "合规校验" | `s3/交叉对比报告.md` | `s5/合规报告.md` | 对照招标要求逐条检查有无漏项或不合规 |

##### ④ 信息恢复

所有 AI 处理完成后，把占位符替换回真实信息。

- **纯本地处理，不调用任何 AI 模型**——读取工作目录 `desensitize/mapping.json` 中的脱敏映射，自动将输出文件中的占位符（如 `<!-- 000001 -->`）替换回原始敏感信息
- 页面展示恢复前后的对比（高亮标记已恢复的位置）
- 用户确认恢复结果是否正确
- 用户手动编辑占位符导致恢复失败时：高亮标记失败项，允许用户手动修正
- 确认后：恢复后的完整文档写入对应步骤子目录，流转给下游节点（文件导出），确保人看到的是真实内容

##### ⑤ 文件导出

选择格式，预览并导出下载。

- 选择导出格式（Word .docx / PDF / Excel .xlsx / Markdown .md）
- 选择导出排版模板（如公司标准 Word 模板）
- 预览导出效果
- 设置文件名
- 可选择导出哪些内容（如果流程中有多个内容产出）
- 点击「导出」下载文件，导出文件同时存储在工作目录的 `export/` 子目录中
- 确认后：文件生成并下载，流程结束

**管理员可配置项：**

| 配置项 | 说明 |
| ---- | ---- |
| 可用导出格式 | 该流程支持的导出格式列表 |
| 排版模板 | 关联的导出排版模板文件 |
| 内容映射规则 | 定义文档内容如何映射到导出模板的各个区域 |
| 导出目录 | 导出文件在工作目录中的存储子目录（默认 `export/`） |

##### 暂缓节点类型（后续版本支持）

以下节点类型在后续版本中支持：

| 暂缓节点 | 说明 |
| ------ | ---- |
| 选择资料 | 从项目资料库检索参考资料，资料内容通过 RAG 检索后作为上下文传递给下游 AI 节点 |
| 大纲确认 | AI 先生成文档大纲/目录结构，用户确认调整后再生成正文 |
| 人工审核 | 指定审核人审核内容，通过或打回，支持审批流转和通知 |

##### 每个节点的通用操作能力

所有节点除了各自的专属功能外，均支持以下四种通用操作：


| 操作         | 说明                                                                                      |
| ---------- | --------------------------------------------------------------------------------------- |
| **确认，下一步** | 正常流转到流程中的下一个节点                                                                          |
| **编辑当前内容** | 展开内联编辑器（Markdown编辑+预览），当场修改当前节点输出文件的内容后再继续。支持 AI 辅助编辑（信息恢复前可用线上模型，恢复后仅限本地模型）                |
| **跳过后续**   | 跳过后续链路中的**可选节点**，自动流转至最近的下一个必经节点（如信息恢复）；若后续无必经节点则直接进入文件导出。跳过的可选节点标记为"已跳过" |
| **回退**     | 回到之前的某个节点重新执行，后续节点的输出文件保留为历史版本，状态重置为"待执行"。用户不满意 AI 生成结果时可直接回退重新生成，实现迭代                               |


> 通过这四种通用操作，用户可以在任意节点灵活地编辑内容、跳过不需要的步骤、回退重做——无需专门的"人工编辑"或"选择下一步"节点。

#### 2.4.6 节点间数据流

- 上游节点的输出文件自动作为下游节点的可用输入（通过文件路径引用）
- 多个上游节点的输出可在提示词中同时引用（管理员配置引用哪些步骤目录的哪些文件）
- 节点配置中的变量占位符由系统自动替换为实际文件路径
- 下游节点通过读取上游步骤子目录中的文件获取数据，文件格式为节点配置中定义的格式（Markdown / JSON / 纯文本等）

#### 2.4.7 流程引擎后台能力

以下能力由流程引擎自动处理，不作为用户交互节点：


| 能力   | 说明                            |
| ---- | ----------------------------- |
| 条件路由 | 管理员配置的自动条件分支（如"字数>5000走长文流程"） |
| 版本快照 | 每个节点完成后自动保存版本记录（对应步骤子目录的文件快照）              |
| 通知推送 | 特定节点完成时自动发送通知（可配置在任意节点上）      |
| 并行执行 | 管理员配置的并行分支（如多个模型同时调用，各自输出到同一步骤目录的不同文件）         |
| 工作目录管理 | 自动创建/清理工作目录，管理文件生命周期          |

### 模块五：AI 模型配置

#### 2.5.1 模型提供商（Provider）管理

支持多种模型提供商接入（如阿里云 DashScope、OpenAI 兼容接口、自定义 HTTP 接口等），具体 API 协议详见附录 B。

Provider 实例管理：

- 创建 Provider 实例（选择类型、填写名称、API 地址、认证信息）
- 编辑 Provider 实例
- 测试连通性（发送测试请求验证配置是否正确）
- 启用 / 停用
- 删除

#### 2.5.2 模型管理

每个 Provider 实例下可配置多个模型：

- 添加模型（模型 ID、显示名称、描述、部署类型、**调用方式**、参数配置）
- 编辑模型信息
- 启用 / 停用模型
- 删除模型
- **部署类型**（必填）：


| 部署类型 | 说明                           | 可处理敏感信息 |
| ---- | ---------------------------- | ------- |
| 线上云端 | 调用外部云端 API（如阿里云 DashScope 等） | ✗       |
| 本地私有 | 部署在公司内网的模型服务                 | ✓       |


- 信息脱敏节点：仅可使用**本地私有**模型（因需处理敏感原文）
- 信息恢复节点：纯本地处理（读取数据库中的脱敏映射进行替换），不调用 AI 模型
- 模型调用节点：在脱敏段内（信息脱敏和信息恢复之间），线上/本地均可（提示词中的脱敏规则仅包含类型描述，不含真实值）；在脱敏段外，线上/本地均可
- 各节点"当场编辑"功能的 AI 辅助：在信息恢复之后，仅可使用**本地私有**模型
- **调用方式**（v4 新增，必填）：

| 调用方式 | `invoke_type` | 版本支持 | 配置项 |
| ------- | ------------- | ------- | ----- |
| CLI 命令行 | `cli` | v1（首选） | 命令行调用模板（如 `claude -p "{{提示词}}"`） |
| API 直接调用 | `api` | v2（扩展） | API 地址、认证信息（复用 Provider 配置） |

- **CLI 命令行调用模板**（v1）：
  - 定义该模型的命令行调用格式（如 `claude -p "{{提示词}}"` 或 `glm-cli --model glm-5 --prompt "{{提示词}}"` 等）
  - 不同模型可有不同的命令行调用方式
  - 模板中支持变量插值（`{{提示词}}`、`{{模型参数}}` 等）
- **API 调用配置**（v2，后续扩展）：
  - 复用 Provider 实例的 API 地址和认证信息
  - 配置请求格式映射（提示词→API 字段的映射规则）
  - 配置响应解析规则（API 返回→输出内容的提取规则）
  - 支持流式响应（SSE）和非流式响应
- 模型参数配置（两种调用方式通用）：
  - temperature（创造性）
  - max_tokens（最大输出长度）
  - top_p
  - 其他模型特定参数（以 JSON 形式扩展）

#### 2.5.3 用量与限制

- 按用户 / 部门 / 项目 / 模型设置每日 / 每月调用次数限制
- 按模型设置单次最大 Token 限制
- 用量统计面板（调用次数、Token 消耗，按模型 / 用户 / 部门 / 项目维度）
- 超限提醒与处理策略（提醒 / 阻断 / 允许超限）

### 模块六：文档工作台

> 用户日常操作的核心界面。用户按流程逐节点执行，每个节点是一个操作页面。

#### 2.6.1 我的待办

系统全局入口，汇聚当前用户**跨项目**的所有待处理任务：

- **进行中**：当前用户正在执行流程但尚未完成的文档

每条待办显示：文档标题、所属项目、文档类型、当前节点、创建时间。按最后操作时间排序，支持筛选。

点击待办直接进入对应文档的当前节点操作页面。

> 人工审核相关的待办分类（待审核、被驳回）将在后续版本支持。

#### 2.6.2 创建文档

在项目内新建文档，支持以下三种入口：

- **从空白创建**（默认方式）：
  1. 选择文档类型（从已启用的类型列表中选择）
  2. 选择流程（展示该文档类型下的可用流程，默认流程标记"推荐"，每个流程附带一句话适用场景说明）
  3. 填写文档基本信息（文档标题、描述/备注）
  4. **系统自动创建工作目录**（在服务器指定根目录下按规则创建本次生成的目录结构）
  5. 进入文档工作台
- **从已有文档导入**：上传 Word / PDF 文件，系统解析内容后进入流程，用户可选择从哪个节点开始执行
- **从历史文档复制**：基于项目内已有文档创建副本（复制工作目录），在副本上继续加工

#### 2.6.3 工作台界面

- **流程进度导航栏**（顶部）：展示当前流程的所有节点，标记状态（已完成 ✓ / 进行中 → / 待执行 ○）
- **当前节点操作区**（主体）：根据节点类型展示对应的操作界面（参见 2.4.5 各节点描述）
- **节点历史面板**（侧边）：查看已完成节点的输入输出文件记录
- 可回退到已完成的节点查看历史或重新执行（重新执行后，后续节点状态重置为"待执行"，旧输出文件保留为历史版本）

#### 2.6.4 流程执行

- 用户在每个节点页面上完成操作后，点击「确认/下一步」流转到下一个节点
- 用户在任意节点可点击「回退」回到之前的节点重新执行，实现迭代
- 到达「文件导出」节点时，用户可下载文件，流程结束
- 支持多模型流式输出（技术协议详见附录 A）

#### 2.6.5 失败恢复与任务连续性

- **自动保存**：每个节点的操作过程自动保存草稿（表单输入、编辑内容），写入工作目录，无需用户手动保存
- **断点续作**：浏览器关闭或网络中断后，重新进入文档可恢复到上次操作状态（基于工作目录中的文件状态恢复）
- **后台执行**：AI 生成支持后台执行（CLI 模式为后台进程，API 模式为异步请求），用户可离开页面，生成完成后通过企业微信消息和系统内通知提醒用户
- **部分失败处理**：多模型生成时，已成功的模型输出文件保留，失败的可单独重试
- **取消生成**：支持取消正在进行的 AI 生成任务（CLI 模式终止进程，API 模式取消请求）

#### 2.6.6 通知规则

- 默认通知场景：
  - 后台生成完成：AI 后台生成完成时，通知文档创建人
- 通知渠道：企业微信消息 + 系统内待办列表

> 人工审核相关通知场景（待审核通知审核人、被驳回通知创建人）将在后续版本支持。

#### 2.6.7 全局工作台增强（后续迭代）

> 以下能力在后续版本中逐步支持：

- 全局搜索：跨项目搜索文档、资料、流程
- 最近访问：最近打开的文档列表
- 收藏：收藏项目、文档、常用流程
- 常用流程快捷创建：一键使用常用流程新建文档

### 模块七：文档管理

#### 2.7.1 文档列表

- 项目内文档列表（按创建时间 / 类型 / 状态 / 使用流程 筛选和排序）
- 列表展示信息：文档标题、文档类型、使用流程、创建人、创建时间、当前节点、状态
- 文档搜索（按标题、描述关键词搜索）
- 文档状态：草稿 / 流程执行中 / 已完成
- 批量操作（批量删除，删除后进入回收站）

#### 2.7.2 文档详情

- 查看文档基本信息（标题、类型、使用流程、创建人、创建时间、最后更新时间）
- **工作目录浏览**：可查看文档对应的工作目录结构和各步骤子目录中的文件
- **流程执行历史**：展示每个节点的执行记录，每条记录可查看：
  - 该节点的配置（节点类型、显示名称、提示词模板、模型选择等）
  - 用户的输入内容
  - 节点的输出结果文件（可直接查看文件内容）
  - 使用的 AI 模型（如有）
  - 执行耗时
  - 调用日志（CLI 命令行日志或 API 请求/响应日志，用于调试，管理员可查看）
- 如流程有迭代（回退重新执行），展示迭代历史（迭代 1 / 迭代 2 / ...），可切换查看

#### 2.7.3 内容编辑

- 从流程中任意模型调用节点的输出文件中，选定一个版本标记为「最终版本」
- 基于最终版本可进行两种编辑方式：
  - **手动编辑**：在 Markdown 编辑器中直接修改内容，支持实时预览
  - **AI 辅助修改**：输入修改提示词，选择某个模型，AI 基于当前内容进行修改，流式输出修改结果
- AI 辅助修改的结果可选择「采纳」（替换最终版本）或「放弃」
- 每次编辑保存都生成新的版本记录，可回溯

#### 2.7.4 版本管理

- 流程中每个节点完成时自动生成版本快照（对应步骤子目录的文件快照）
- 用户在任意节点"当场编辑"保存也生成新版本
- 版本列表查看（时间线形式）
- 版本对比（选择两个版本进行 Diff 查看）
- 版本回退（将某个历史版本恢复为当前版本）

#### 2.7.5 导出

AI 生成的内容统一为 Markdown 格式，导出时转换为目标格式：

- 导出为 Word 文档（`.docx`）
- 导出为 Excel 表格（`.xlsx`）
- 导出为 PDF 文件（`.pdf`）
- 导出为 Markdown 文件（`.md`，原始格式）
- 可选择导出 AI 原始输出或用户编辑后的最终版本
- 批量导出（选择多个文档打包下载）

#### 2.7.6 文档可见性

文档创建人可设置文档的可见范围：


| 可见范围     | 说明                          |
| -------- | --------------------------- |
| 仅自己可见    | 只有文档创建人可查看和编辑（**新建文档默认选项**） |
| 项目成员可见   | 项目内所有成员可查看                  |
| 指定成员可见   | 仅创建人指定的项目成员可查看              |
| 指定项目外成员可见 | 可将文档分享给项目外的系统用户（通过企业微信通讯录搜索指定），被分享者可查看该文档并在自己的项目中将其作为参考资料引用 |


- **项目负责人**始终可查看项目内所有文档，不受可见性设置限制
- 文档可见性定义**查看范围**；文档的编辑、继续生成、提交审核等操作仅限**文档创建人**和**项目负责人**执行，其他可见成员默认为只读

> 文档审核锁定和复审规则将在人工审核节点支持后生效。

#### 2.7.7 文档评论与批注（后续版本）

> 后续版本支持文档评论与行内批注能力，实现更轻量的协作反馈。

## 三、核心使用流程

### 3.1 管理员初始化流程

```
管理员登录系统
    │
    ├── 1. 配置模型提供商与模型
    │   ├── 添加 Provider 实例（如 DashScope）
    │   ├── 填写 API 地址和认证信息
    │   ├── 测试连通性
    │   ├── 在 Provider 下添加具体模型（标注部署类型：线上云端/本地私有）
    │   └── 配置模型的命令行调用模板
    │
    ├── 2. 创建文档类型
    │   └── 创建文档类型（名称、编码、描述）
    │
    ├── 3. 在文档类型下创建流程
    │   ├── 选择所属文档类型
    │   ├── 填写流程名称和描述
    │   ├── 从空白开始编排节点：
    │   │   ├── 添加输入转换节点，配置输入表单字段和输出文件定义
    │   │   ├── 可选：添加信息脱敏节点，配置脱敏规则和占位符格式
    │   │   ├── 添加模型调用节点（可多个），逐个配置：
    │   │   │   ├── 节点显示名称和步骤说明
    │   │   │   ├── 输入文件配置（读取哪些上游目录/文件）
    │   │   │   ├── 提示词模板（含 {{脱敏规则}} 等系统变量）
    │   │   │   ├── 可用模型范围和命令行调用模板
    │   │   │   └── 输出文件定义（目录、文件名）
    │   │   ├── 可选：添加信息恢复节点（与脱敏成对）
    │   │   └── 添加文件导出节点
    │   ├── 系统自动校验流程合理性
    │   ├── 设为该文档类型的默认流程（可选）
    │   └── 启用流程
    │
    └── 4. 用户管理（企业微信自动同步）
```

### 3.2 用户日常使用流程

```
用户登录系统
    │
    ├── 1. 进入或创建项目
    │   ├── 从「我的项目」列表选择已有项目
    │   └── 或创建新项目，邀请相关成员
    │
    ├── 2. 上传项目资料（可在任意时间进行）
    │   ├── 进入项目资料库
    │   ├── 上传相关参考文件，设置资料分类
    │   └── 系统自动解析、RAG 处理，等待状态变为「就绪」
    │
    ├── 3. 新建文档
    │   ├── 选择文档类型
    │   ├── 选择流程（默认推荐 / 其他可用）
    │   ├── 填写文档标题
    │   ├── 系统自动创建工作目录
    │   └── 进入文档工作台
    │
    ├── 4. 按流程逐节点执行
    │   ├── 每个节点是一个操作页面，输入输出均为工作目录中的文件
    │   ├── 完成操作后点击「确认/下一步」流转
    │   ├── 可在任意节点当场编辑输出文件内容（支持AI辅助）
    │   ├── 不满意时可回退到之前的节点重新执行，实现迭代

    └── 5. 导出
        ├── 在导出节点选择格式并下载
        └── 或在文档管理中随时导出

### 3.3 文档生成的完整数据流（以需求分析为例）

```
输入转换（用户上传需求材料 → 写入 s1/需求清单.md）
    ──→ 信息脱敏（定义映射规则 → 存储 desensitize/mapping.json）
    ┌─── 以下所有AI处理，提示词中自动注入脱敏规则 ─────────────────────┐
    │ ──→ 模型调用[glm-5]:                                            │
    │     命令行: claude -p "...读取 s1/需求清单.md...{{脱敏规则}}...    │
    │            输出到 s2/需求报告-glm5.md"                           │
    │ ──→ 模型调用[kimi-k2.5]:    s1/ → s2/需求报告-kimi-k2.5.md     │（并行）
    │ ──→ 模型调用[MiniMax-M2.5]: s1/ → s2/需求报告-MiniMax-M2.5.md  │
    │ ──→ 模型调用[deepseek-v3.2]:s1/ → s2/需求报告-deepseek-v3.2.md │
    │                                                                  │
    │ ──→ 模型调用[qwen3.5-plus]: 交叉对比                             │
    │     命令行: claude -p "...读取 s2/*.md...{{脱敏规则}}...          │
    │            输出到 s3/交叉对比-需求分析报告.md"                     │
    │     ※ 用户可在此处回退重新生成，实现迭代                            │
    └──────────────────────────────────────────────────────────────────┘
    ──→ 信息恢复（读取 mapping.json，<!-- 000001 --> → 真实值）
    ──→ 文件导出（导出 Word → export/最终报告.docx）
```

```
项目负责人创建项目

    ├── 邀请团队成员（参与者）、可添加其他负责人

    │
    └── 项目负责人：

        ├── 管理成员、项目配置、资料库
        └── 查看项目统计数据

## 四、非功能需求概述

> 仅列出关键的非功能需求方向，不涉及具体技术实现。


| 维度  | 要求                                                     |
| --- | ------------------------------------------------------ |
| 性能  | 流式生成响应延迟 < 2 秒（首 Token）；支持 50+ 用户同时生成；工作目录文件 I/O 性能保障                  |
| 安全  | 数据传输加密；API Key 等敏感信息加密存储；接口鉴权；脱敏映射数据库加密存储不上传给线上AI；脱敏节点仅可调用本地私有模型；工作目录访问权限隔离（用户只能访问自己有权限的文档对应的工作目录） |
| 可用性 | 单模型失败不影响其他模型；单节点失败不影响流程中其他节点；CLI 进程异常可自动重启、API 调用超时可重试；服务异常可快速恢复                 |
| 数据  | 生成记录完整保留；流程执行历史可审计；工作目录文件长期归档；支持数据备份与恢复                           |
| 易用性 | 界面简洁直观，核心操作路径不超过 3 步；流程编排提供可视化拖拽界面                     |

| 存储  | 工作目录磁盘空间管理（已完成文档可归档压缩）；定期清理策略（如超过 N 天的已完成文档工作目录可自动归档） |

## 五、关键业务规则

> 集中说明产品中的关键边界规则，避免研发和业务理解不一致。


| #   | 规则             | 说明                                                   |
| --- | -------------- | ---------------------------------------------------- |
| 1   | 文档不支持多人同时编辑    | 同一文档同一时间只允许一个用户编辑，其他人只能查看                            |
| 2   | 审核中文档锁定（后续版本） | 文档进入人工审核节点后，内容锁定不可编辑，需审核人驳回后创建人才能修改。*人工审核节点支持后生效*   |
| 3   | 已完成文档再编辑触发复审（后续版本） | 已完成的文档如需再编辑，状态自动变为「待复审」，需重新走审核流程。*人工审核节点支持后生效*  |
| 4   | 删除统一走回收站       | 文档和资料的删除均为软删除，进入回收站，项目负责人可恢复                         |
| 5   | 项目删除为软删除       | 项目删除后其文档和资料保留，系统管理员可恢复                               |
| 6   | 支持文档另存为        | 用户可将文档「另存为新文档」，在副本上继续修改（复制工作目录），不影响原文档                       |
| 7   | 流程变更不影响进行中文档   | 管理员修改流程后，已在执行中的文档继续使用创建时的流程快照（包括节点配置、提示词、模型选择等），不受影响 |
| 8   | 跨项目不可复制文档，但可引用 | 文档与项目绑定，不支持跨项目复制；但项目负责人可跨项目引用资料，被分享的文档可被项目外成员作为参考资料只读引用 |
| 9   | 资料引用版本固定       | 文档在流程执行中引用资料时，固定引用到当时的资料版本快照；资料后续更新不影响历史文档的引用结果与审计追溯 |
| 10  | 项目负责人退出须移交权限   | 项目负责人不可直接退出项目，必须先将负责人角色移交给其他成员；唯一负责人必须先指定新负责人 |
| 11  | 同一节点类型可多次出现    | 流程中同一基础节点类型可多次添加，各实例独立配置，互不影响 |
| 12  | 流程与文档类型绑定      | 每个流程归属于特定文档类型，用户新建文档时只能选择该文档类型下的流程 |
| 13  | 工作目录与文档绑定      | 每个文档对应唯一的工作目录，文档删除时工作目录进入归档（不立即物理删除） |
| 14  | 脱敏真实值不出服务器     | 脱敏映射中的真实值（`original` 字段）加密存储在数据库中（工作目录 `desensitize/mapping.json` 保留本地副本），注入到提示词中的脱敏规则仅包含类型描述，不包含真实值 |
| 15  | 调用方式对流程透明      | 模型的调用方式（CLI/API）对上层流程编排透明，切换调用方式不影响流程配置和节点行为；v1 首选 CLI 命令行，v2 扩展 API 直接调用 |

## 附录

### 附录 A：SSE 事件协议

模型调用节点中多模型生成的流式通信协议：


| 事件类型          | 说明       | 数据              |
| ------------- | -------- | --------------- |
| `model_start` | 某模型开始生成  | 模型 ID           |
| `token`       | 流式内容片段   | 模型 ID + 内容文本    |
| `model_done`  | 某模型生成完成  | 模型 ID + 输出记录 ID + 输出文件路径 |
| `model_error` | 某模型生成失败  | 模型 ID + 错误信息    |
| `all_done`    | 全部模型生成完成 | 无               |


### 附录 B：Provider 类型与 API 风格


| Provider 类型   | 说明                     | API 风格                  |
| ------------- | ---------------------- | ----------------------- |
| 阿里云 DashScope | 阿里旗下模型网关               | OpenAI 兼容接口             |
| OpenAI 兼容     | 兼容 OpenAI API 格式的第三方服务 | OpenAI Chat Completions |
| 自定义 HTTP      | 自定义 HTTP 接口            | 自定义请求/响应映射              |
| 命令行 CLI       | 通过命令行方式调用的模型（v4 新增，v1 首选）    | 命令行调用模板 + stdout 输出     |


每种 Provider 类型为系统内置代码实现，封装了：

- API 请求格式构建 / 命令行参数组装
- 流式响应解析 / stdout 流式读取
- 错误处理与重试策略
- 认证方式（API Key / Token / 命令行认证等）

> **调用方式与 Provider 的关系**：
> - CLI 命令行（v1）：每个模型独立配置命令行模板，不强依赖 Provider 的 API 地址配置
> - API 直接调用（v2）：复用 Provider 实例的 API 地址、认证信息和协议适配，模型只需配置模型 ID 和参数
> - 同一 Provider 下的不同模型可以使用不同的调用方式（如部分模型用 CLI，部分用 API）

### 附录 C：工作目录结构规范（v4 新增）

每次文档生成的工作目录遵循以下结构规范：

```
{根目录}/{项目编码}/{文档ID}/{时间戳}/
├── meta.json                    ← 本次生成的元信息（流程快照、创建时间、状态等）
├── s1/                          ← 步骤1（输入转换）输出
│   ├── 需求清单.md              ← 节点配置中定义的输出文件
│   └── _raw/                    ← 用户上传的原始文件（解析前）
├── s2/                          ← 步骤2（模型调用）输出
│   ├── 需求报告-glm5.md         ← 各模型的输出文件
│   ├── 需求报告-kimi-k2.5.md
│   └── _logs/                   ← 命令行调用日志
│       ├── glm5.log
│       └── kimi-k2.5.log
├── s3/                          ← 步骤3（交叉对比）输出
│   └── 交叉对比-需求分析报告.md
├── desensitize/                 ← 脱敏相关（本地存储，不传给AI）
│   └── mapping.json             ← 敏感信息映射表
├── export/                      ← 最终导出文件
│   └── 最终报告.docx
└── _history/                    ← 历史版本（回退重做时旧文件归档于此）
    ├── s2_v1/
    └── s2_v2/
```

**目录命名规则**：
- `s{N}/` — 按节点在流程中的顺序编号
- `_raw/` — 存储用户上传的原始文件
- `_logs/` — 存储命令行调用的执行日志
- `_history/` — 存储回退重做时的历史版本文件
- `desensitize/` — 固定目录名，存储脱敏映射
- `export/` — 固定目录名，存储导出文件
