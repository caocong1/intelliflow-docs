**packages/backend/src/modules/runtime/model-call.service.ts:**

1. Add `resolveFieldPath()` helper function for deep JSON field access:
   - Parse fieldPath string (e.g. "items[0].name") into segments array
   - Support string keys, numeric indices, and `[*]` traversal marker
   - For `[*]` traversal: map over array elements, recursively resolve remaining path, return JSON.stringify of results array
   - Return `string | undefined`

2. Extend `resolveRef()` to handle `ref.fieldPath`:
   - After finding base value via existing 6-level priority chain, if `ref.fieldPath` is present:
   - For namedOutputs and models: attempt `JSON.parse(value.content)`, then call `resolveFieldPath(parsed, ref.fieldPath)`
   - For other levels: attempt `JSON.parse(String(baseValue))`, then `resolveFieldPath`
   - If JSON.parse fails, return undefined (with console.warn)
   - When no fieldPath: existing auto-unwrap behavior for namedOutputs/models returning `.content` is already correct

3. Extend `resolvePromptTemplate()` to handle multi-level fieldPath in `{{nodeId.segmentKey.field.path}}` syntax:
   - After extracting nodeId and the first dot-separated segment as segmentKey, check if there are additional segments
   - If so, join remaining segments as fieldPath and pass to resolveRef
   - Example: `{{n3.clause_list.items[0].name}}` → nodeId="n3", segmentKey="clause_list", fieldPath="items[0].name"

4. Add `validateModelOutput()` function:
   - Takes content string and ModelCallConfig
   - If `outputFormat !== "json"`, return `{ status: "completed" }`
   - Layer 1: `JSON.parse(content)` — on fail return `{ status: "format_error", errors: ["JSON 语法错误: {message}"] }`
   - Layer 2: if `jsonSchema` provided, validate with ajv (allErrors: true). On fail return `{ status: "format_error", errors: schemaErrors }`
   - On success return `{ status: "completed" }`

5. Add `parseNamedOutputs()` function:
   - Takes raw content string and expected IDs array
   - Use regex `/===OUTPUT:(\w+)===\n?([\s\S]*?)===END:\1===/g` to extract named segments
   - If all expected IDs found: return `{ namedOutputs: Record<id, { content, format }>, fallback: false }`
   - Otherwise: return `{ namedOutputs: { _default: { content: rawContent, format: "text" } }, fallback: true }`
   - The format for each named output comes from the matching NamedOutputDef.format

6. Integrate validation + parsing into the model output completion flow:
   - After model streaming completes and full content is available, call `validateModelOutput()` if `outputFormat === "json"`
   - If config has `namedOutputs`, call `parseNamedOutputs()` and store result in `outputData.namedOutputs`
   - Store `fallbackWarning: true` in outputData when fallback triggered
   - Store `formatErrors` in outputData when format_error status

7. Add prompt injection in `resolvePromptTemplate()`:
   - After existing desensitize rule injection, add JSON Schema injection:
     ```
     if (config.jsonSchema) {
       resolved += `\n\n请严格按照以下 JSON Schema 格式输出：\n\`\`\`json\n${JSON.stringify(config.jsonSchema, null, 2)}\n\`\`\``;
     }
     ```
   - Add named output delimiter injection:
     ```
     if (config.namedOutputs?.length) {
       const format = config.namedOutputs.map(o => `===OUTPUT:${o.id}===\n[${o.name}内容]\n===END:${o.id}===`).join("\n\n");
       resolved += `\n\n请按以下格式分段输出，每个产物用指定分隔符包裹：\n${format}`;
     }
     ```
   - Note: `resolvePromptTemplate` needs the ModelCallConfig passed in. Add config parameter or extract injection into a separate function called from the route handler after prompt resolution.
