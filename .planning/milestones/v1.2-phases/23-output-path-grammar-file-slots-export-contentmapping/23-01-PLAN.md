**derive-outputs.ts changes:**

Rewrite `deriveOutputs` to produce segmentKey-aware outputs with type-prefixed IDs:

- `input_transform` case:
  - Text/textarea fields: `id: ${nodeId}-field-${key}` where `key = field.machineKey || field.id`, `segmentKey: key`
  - File fields WITH fileSlotId: `id: ${nodeId}-fileslot-${field.fileSlotId}`, `segmentKey: field.fileSlotId`, name from fileSlotLabel
  - Keep merged "file upload" output: `id: ${nodeId}-file-upload`, `segmentKey: "text"` for backward compat
  - Also generate outputs for number/date/datetime/select/multiselect fields (same pattern as text: `field-` prefix, machineKey or id as key)

- `model_call` case: add `segmentKey: modelId` to each output (trivial one-line addition)

- `desensitize` case: add `segmentKey` derived from outputId suffix (e.g., the src.outputId part)

- `restore` case: add `segmentKey` derived similarly

- `export` case: unchanged (returns empty array)

Follow the exact code pattern from 23-RESEARCH.md "derive-outputs.ts -- Updated Input Transform Case" section.

**validation.ts changes:**

2. **Update Rule 7 (broken variable references):** Change `outputIdSet` construction from `${n.id}.${o.id}` to `${n.id}.${o.segmentKey || o.id}`. This ensures variable references using segmentKey format can be validated.

3. **Update contentMapping check** (same block as Rule 7): The `outputIdSet.has(...)` check already uses `${ref.nodeId}.${ref.outputId}`. Since `ref.outputId` now stores segmentKey, and the set now uses segmentKey, these align automatically.
