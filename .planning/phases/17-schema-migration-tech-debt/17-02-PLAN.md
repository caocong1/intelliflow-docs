**In document-types.service.ts:**

1. Find the DTYPE-04 TODO (around line 134) in the `deleteDocumentType` function.

2. Add a function to query associated documents for a document type. The relationship is INDIRECT — documents reference workflows, workflows reference document types:
```typescript
// Query: SELECT d.id, d.title FROM documents d
//        INNER JOIN workflows w ON d.workflow_id = w.id
//        WHERE w.document_type_id = $documentTypeId
//        AND d.is_deleted = false
```
Use Drizzle's `.select().from(documents).innerJoin(workflows, eq(documents.workflowId, workflows.id)).where(and(eq(workflows.documentTypeId, id), eq(documents.isDeleted, false)))` pattern.

3. Modify the delete guard logic to check BOTH workflows AND documents. The existing code already checks workflows — add the documents check alongside it. If either has results, throw an error with BOTH lists attached.

4. Change the error structure: throw an error with message `"HAS_ASSOCIATIONS"` and attach an `associations` property containing `{ workflows: [{id, name}], documents: [{id, title}] }`.

5. Ensure soft-deleted documents (isDeleted=true) are excluded from the check.

**In document-types.routes.ts:**

1. Find the existing error handler for `HAS_ASSOCIATED_WORKFLOWS`.

2. Replace/extend it to handle the new `HAS_ASSOCIATIONS` error. Return a 409 response with this structure:
```json
{
  "error": "无法删除：该文档类型存在关联的工作流或文档",
  "workflows": [{"id": "...", "name": "..."}],
  "documents": [{"id": "...", "title": "..."}]
}
```

3. Keep backward compatibility — if only workflows exist (no documents), the documents array should be empty `[]`, and vice versa.

**Important:** The document-to-documentType relationship is INDIRECT through workflows. Do NOT look for a documentTypeId column on the documents table — it does not exist. Always join through workflows.
