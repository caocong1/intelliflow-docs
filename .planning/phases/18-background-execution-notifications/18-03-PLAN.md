**A) Add API helper** in `api.ts` (or wherever API calls are centralized):
   - `startBackgroundExecution(documentId: string)` → POST `/runtime/${documentId}/start-background`

**B) Replace frontend-driven execution flow in DocumentWorkspace.tsx**:

The current flow drives each node step from the frontend. Replace with:

1. **Start generation**: When user confirms input and triggers "generate", call `startBackgroundExecution(documentId)` instead of the current step-by-step flow. This returns immediately with `{ status: "queued" }`.

2. **Polling for progress**: After starting background execution, set up a polling interval:
   ```typescript
   const WORKSPACE_POLL_INTERVAL = 3; // seconds
   const [countdown, setCountdown] = createSignal(WORKSPACE_POLL_INTERVAL);
   const [isGenerating, setIsGenerating] = createSignal(false);

   // Countdown + poll loop
   const pollTimer = setInterval(() => {
     setCountdown(prev => {
       if (prev <= 1) {
         fetchRuntimeState(); // poll backend
         return WORKSPACE_POLL_INTERVAL;
       }
       return prev - 1;
     });
   }, 1000);

   onCleanup(() => clearInterval(pollTimer));
   ```

3. **State recovery on page load**: On component mount, call `GET /runtime/:documentId` to get current state. If any node has status `in_progress` or document status is `in_progress`, start polling automatically. If all nodes are completed/failed, no polling needed.

4. **UI updates from polled state**: Map the polled `DocumentRuntimeState` to the existing workspace UI:
   - Completed nodes: show output data (same as current)
   - In-progress nodes: show loading spinner/animation
   - Failed nodes: show red status indicator + errorMessage summary + retry button
   - Pending nodes: show as waiting (same as current)

5. **Countdown display**: Show a small indicator near the workspace header: "X秒后自动刷新" + a manual refresh icon button. Only visible during active generation (isGenerating = true).

6. **Stop polling**: When all nodes reach terminal state (completed/failed/skipped) or document status is completed/failed, clear the interval and hide the countdown.

**C) Failed node retry UX**:
   - Failed nodes display: red border/background, error message text (e.g., "模型调用超时"), retry button
   - Retry button calls `startBackgroundExecution(documentId)` which the backend handles by resuming from the failed node (existing rollbackToNode + re-execute pattern, or the `startFromNodeId` parameter)
   - After retry, restart polling

**D) Keep existing workspace UI structure** mostly unchanged — the node output viewers, editors, model comparison panels all stay the same. The only change is the execution trigger and status data source (polling vs frontend-driven).
