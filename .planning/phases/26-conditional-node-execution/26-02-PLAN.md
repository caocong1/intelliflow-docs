Props interface:
```typescript

  rule: NodeExecutionRule | undefined;
  upstreamNodes: FlowNodeData[];

  onChange: (rule: NodeExecutionRule | undefined) => void;

```

Component structure (collapsible panel, per user decision):

1. **Collapsible header**: "执行条件" label + expand/collapse chevron icon + condition count badge (if conditions exist). Collapsed by default when `rule` is undefined. When expanded and no rule exists yet, show "添加执行条件" button.

2. **Action selector** (shown when rule exists): Two radio-style buttons for "跳过" and "阻断". No default selected — user must pick one (per user decision). Include brief descriptions: "跳过：条件满足时自动跳过此节点" and "阻断：条件满足时停止流程".

3. **Logic selector** (shown when >1 condition): Toggle between "全部满足 (AND)" and "任一满足 (OR)".

4. **Condition rows** (For each condition in the array):
   - **Variable selector**: Render `VariablePicker` component (reused, per user decision) for `sourceRef`. Display selected variable name as a clickable chip/button that opens the picker. Show "选择变量" placeholder when not set.
   - **Operator dropdown**: `<select>` with options: `等于 (equals)`, `不等于 (not_equals)`, `存在 (exists)`, `不存在 (not_exists)`, `包含 (contains)`.
   - **Value input**: Text input for the comparison value. **Hidden entirely** (not just disabled) when operator is `exists` or `not_exists` (per Claude's Discretion from research). When switching from equals to exists, clear the value field. Simple text input — no smart suggestion for now (simple approach per research recommendation; can enhance later).
   - **Remove button**: "x" icon to remove the condition row.

5. **Add button**: "+ 添加条件" at bottom of condition list.

6. **Remove rule button**: "删除执行条件" link/button to clear the entire rule (calls `onChange(undefined)`).

Styling: Use inline styles consistent with other config panel components (RuntimeSettings, ModelCallConfig patterns). Gray border for the collapsible section. Compact row layout.

State management: Use `createSignal` for expanded state. The `rule` prop is the source of truth — all changes call `onChange()` with the updated rule object (create new object, don't mutate). When adding the first condition, initialize the rule with `{ action: undefined as any, conditions: [emptyCondition], logic: "and" }` — the action being unset signals to the user they must pick one.

Handle edge case: If user removes all conditions, also remove the rule entirely (call `onChange(undefined)`).
