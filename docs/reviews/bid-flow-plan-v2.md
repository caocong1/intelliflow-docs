# 投标文件通用生成流程 v2

> 基于 v1 跨项目对比分析 + GPT 评审反馈，结合 IntelliFlow v4 系统架构约束修订。

---

## 一、GPT 评审意见采纳情况

| # | GPT 建议 | 采纳 | 理由 |
|---|---------|------|------|
| 1 | 输入前提过窄，招标文件不应必选 | **部分采纳** | 拓宽输入材料类型，但不拆"两个入口模式"——"样本沉淀"本质就是往资料库上传，系统已有此功能 |
| 2 | 新增"阶段0：材料接入与归一化" | **不采纳** | IntelliFlow 的「输入转换」节点已内置文件解析（Word/PDF 提取文字、图片 OCR、音视频转文字），不需要额外阶段。大文档切片是系统基础能力，不是流程节点 |
| 3 | 程序性文件不应一刀切为"AI 无法生成" | **采纳** | 拆分为三类：规则填充型 / 证据挂载型 / 人工确认型 |
| 4 | 条款-响应-证据闭环模型 | **部分采纳** | 概念有价值，但不建正式数据模型（ClauseItem/ResponseItem 等），而是让模型调用节点输出结构化 JSON 包含证据引用，质检节点验证证据完备性 |
| 5 | 输出不应只以 Markdown 为中心 | **采纳** | 模型调用节点已支持"结构化 JSON"输出格式，技术应答表、偏差表等应输出为 JSON/表格，非纯 Markdown |
| 6 | 不应一个大节点一次性生成所有章节 | **采纳** | 拆分为多个模型调用节点，按内容组分步生成 |
| 7 | 质检环节偏轻 | **采纳** | 扩展质检项，增加合规校验 |
| 8 | 项目分类方式需调整 | **采纳** | 删除"简单采购类价值低"的先验判断，改为基于实际内容复杂度评估 |
| 9 | 缺少多源资料融合 | **部分采纳** | 资料库 RAG 已支持多源检索，需明确各类补充材料的用途定义 |
| 10 | 导出缺少卷宗装配 | **采纳** | 文件导出节点需支持多卷宗模式 |

**整体判断**：GPT 的分析很细致，但有"把简单问题复杂化"的倾向。建议 2（材料归一化阶段）和建议 4（正式数据模型）属于企业中间件思路，IntelliFlow 是一个流程编排 + AI 生成的工具平台，不需要这么重的数据建模。核心该做的是：**用好已有的 5 种节点，把流程编排做对**。

## 四、三层内容结构（v2 修订）

### 原版问题

v1 把第一层统一标注为"AI 无法生成，需人工附件/模板填充"，过于粗糙。

### v2 修订

投标文件内容可分为三层，每层的处理策略不同：

```
┌─ 第一层：程序性文件 ──────────────────────────────────────────┐
│                                                                │
│  A. 规则填充型（AI 可基于主数据 + 模板自动组装）              │
│     投标函、授权委托书、承诺函、资格审查表、                  │
│     中小企业声明、商务偏差表（回答"完全满足"）                │
│     → 系统输入：公司主数据 + 项目信息 + 模板                  │
│     → 由模型调用节点按模板规则填充                            │
│                                                                │
│  B. 证据挂载型（需人工上传附件，系统管理索引）                │
│     营业执照、资质证书、审计报告、纳税社保证明、              │
│     检测证书、原厂授权函、业绩合同                            │
│     → 用户在输入转换节点上传                                  │
│     → 系统记录附件清单和位置，供导出时装配                    │
│                                                                │
│  C. 人工确认型（需人签字/盖章/决策，系统无法代替）            │
│     保证金信息、签章页、法人授权链、特殊声明内容              │
│     → 用户手工确认，系统生成占位页提示"此处需签章"           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
┌─ 第二层：技术响应文件（AI + 产品数据）────────────────────────┐
│                                                                │
│  技术偏差/应答表：逐条对照招标文件技术要求回答                │
│  产品参数表：设备型号、规格、配置                              │
│  供货范围表：设备清单 + 数量 + 单价                           │
│                                                                │
│  → 需要：招标文件条款 + 产品选型信息 + 产品资料库             │
│  → 输出格式：结构化 JSON/表格（非纯 Markdown）               │
│  → 条款→响应→证据引用 三要素闭环                              │
│                                                                │
└────────────────────────────────────────────────────────────────┘
┌─ 第三层：方案正文（AI 生成核心价值区）────────────────────────┐
│                                                                │
│  建设/实施方案、供货方案、售后服务方案、质量保证、            │
│  培训方案、运维方案、安全措施、组织架构、进度计划、           │
│  重点难点分析、合理化建议...                                   │
│                                                                │
│  → 需要：招标文件要求 + 产品选型 + 资料库（历史标书等）      │
│  → 按章节组分步生成，非一次性输出                             │
│  → 资料库 RAG 检索增强                                        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 五、通用流程设计（v2 修订）

### 设计原则

1. **严格使用 IntelliFlow 的 5 种节点类型**，不引入系统外概念
2. **多个模型调用节点分步处理**，不做整份一次性生成
3. **每个模型调用节点有明确的单一职责**，方便单步重跑和编辑
4. **用户在每个节点都可编辑和审核**，确保人工把控

### 流程节点编排

```
┌──────────────────────────────────────────────────────────────┐
│ 节点①  输入转换                                              │
│                                                              │
│ 用户操作：                                                    │
│   上传：项目材料（至少一种）                                  │
│     - 招标文件/技术规范书（优先）                             │
│     - 客户需求文档、售前方案（补充）                          │
│     - 产品选型表（如已确定）                                  │
│   填写：项目基本信息（表单）                                  │
│     - 项目名称、招标编号、招标方名称                          │
│     - 投标方名称、法定代表人、授权代理人                      │
│     - 投标截止日期、交货地点                                  │
│   上传：证据附件包（证书、审计报告、业绩合同等）              │
│   引用：资料库中的参考资料                                    │
│     - 公司主数据（简介、资质清单、人员库）                    │
│     - 历史标书（脱敏后）                                      │
│     - 产品技术白皮书/参数手册                                 │
│     - 通用方案模板（售后、培训、质量等）                      │
│                                                              │
│ 系统处理：                                                    │
│   - 自动解析上传文件（Word/PDF 提文字，图片 OCR）            │
│   - 将所有输入整理为结构化文件                                │
│                                                              │
│ 输出 → s1/                                                   │
│   s1/project_info.json         ← 项目基本信息（表单数据）    │
│   s1/tender_doc.md             ← 招标文件/技术规范书解析文本 │
│                                   （如无招标文件则此文件不存在）│
│   s1/product_selection.json    ← 产品选型表（如有）          │
│   s1/supplementary/            ← 补充材料解析文本            │
│   s1/company_snapshot.json     ← 公司主数据快照（从资料库    │
│                                   固化：公司简介、法人信息、  │
│                                   资质清单、人员库摘要）      │
│   s1/evidence_manifest.json    ← 证据附件清单（结构化）      │
│                                   每条含：文件名、分类、      │
│                                   总页数、                    │
│                                   分页摘要数组（每页/每段     │
│                                   的文本摘要+页码锚点）、    │
│                                   关键字段提取（证书编号、    │
│                                   有效期、发证机构、          │
│                                   字段所在页码）              │
│                                                              │
│   evidence_manifest.json 单条示例：                           │
│   {                                                           │
│     "file": "营业执照.pdf",                                   │
│     "category": "资质证书",                                   │
│     "total_pages": 1,                                        │
│     "page_index": [                                          │
│       {"page": 1, "summary": "北京绿色苹果...统一社会信用..."} │
│     ],                                                        │
│     "extracted_fields": {                                     │
│       "cert_no": "91110108...", "cert_no_page": 1,           │
│       "valid_until": "长期", "issuer": "海淀区市场监管局"     │
│     }                                                         │
│   }                                                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐
│ 节点②  信息脱敏                                              │
│                                                              │
│ 本地 AI 辅助识别敏感信息 → 用户确认脱敏映射                  │
│ 脱敏范围：公司名、人名、金额、地址、联系方式等               │
│                                                              │
│ 输出 → desensitize/mapping.json                              │
│ 后续所有模型调用节点的提示词自动注入脱敏规则                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐
│ 节点③  模型调用 [分析招标文件 → 生成投标蓝图]               │
│                                                              │
│ 显示名称：「分析招标要求」                                    │
│ 输入：s1/* （自动检测 tender_doc.md 是否存在）               │
│                                                              │
│ ┌─ 正常模式（有招标文件）─────────────────────────────────┐   │
│ │ 提示词核心任务：                                        │   │
│ │  1. 识别采购方式                                        │   │
│ │     （公开招标/竞争性谈判/政府采购/企业采购）           │   │
│ │  2. 提取评分标准/评审因素表                             │   │
│ │     （权重、分值、评分细则）                             │   │
│ │  3. 提取所有技术要求条款，标记：                         │   │
│ │     - ★ 必须项（不满足则投标无效）                      │   │
│ │     - # 重要项（影响评分）                               │   │
│ │     - 普通项                                             │   │
│ │  4. 识别文件卷宗要求及每章/附件的卷宗归属               │   │
│ │     （单卷 / 资格+商务+技术多卷，                       │   │
│ │      每章标注 volume: 资格|商务|技术）                   │   │
│ │  5. 生成投标文件目录大纲，每章标注：                     │   │
│ │     - 来源依据（招标文件第几条/评分项对应）              │   │
│ │     - 处理方式：规则填充/证据挂载/AI生成/人工确认       │   │
│ │     - 卷宗归属（volume 字段）                            │   │
│ │     - 缺失信息提示（用户需要补充什么）                   │   │
│ │  6. 评估内容复杂度                                       │   │
│ │     （方案正文量、条款密度、证明材料密度）               │   │
│ └──────────────────────────────────────────────────────┘   │
│                                                              │
│ ┌─ 降级模式（无招标文件，仅有需求文档/售前方案等）────┐      │
│ │ 提示词核心任务：                                    │      │
│ │  1. 从已有材料推断项目类型和技术范围                │      │
│ │  2. 基于行业惯例生成建议性章节大纲                  │      │
│ │     （无评分矩阵、无条款清单、无卷宗要求）          │      │
│ │  3. 大纲中所有章节标注为"待确认"                    │      │
│ │  4. 卷宗归属：统一设为 volume: "single"              │      │
│ │     （降级模式不支持多卷导出，仅输出单文件）         │      │
│ │  5. 生成明确警告：缺少招标文件，以下内容为建议性    │      │
│ │     框架，用户必须逐章确认后再继续                  │      │
│ │                                                      │      │
│ │ 限制说明：                                            │      │
│ │  - 无法生成评分矩阵和条款清单                        │      │
│ │  - 技术应答/偏差表无法生成（无条款来源）             │      │
│ │  - 仅支持方案正文和通用章节的生成                    │      │
│ │  - 不支持多卷宗导出（volume 全部为 single）          │      │
│ │  - 适合售前方案编写、历史标书复用等场景              │      │
│ └──────────────────────────────────────────────────┘      │
│                                                              │
│ 输出 → s2/                                                   │
│   s2/bid_blueprint.md          ← 投标蓝图（大纲+处理方式    │
│                                   +卷宗归属volume字段）      │
│   s2/clause_list.json          ← 技术要求条款清单            │
│                                   （降级模式下为空数组）      │
│   s2/scoring_matrix.json       ← 评分标准矩阵                │
│                                   （降级模式下为空对象）      │
│   s2/missing_info.md           ← 缺失信息提示清单           │
│   s2/mode.json                 ← {"mode":"normal"|"degraded"}│
│                                                              │
│ → 用户审核蓝图，编辑调整章节，补充缺失信息                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐
│ 节点④  模型调用 [技术响应 + 规则填充]                        │
│                                                              │
│ 显示名称：「生成技术响应与表格」                              │
│ 输入：s1/* + s2/* + 资料库（产品资料）                       │
│                                                              │
│ 执行前检查 s2/mode.json：                                    │
│   - normal 模式 → 执行全部任务                               │
│   - degraded 模式 → 跳过任务 1/2/4，仅执行任务 3（规则填充）│
│     技术应答和设备清单在降级模式下不生成，输出文件为空        │
│                                                              │
│ 提示词核心任务：                                              │
│   1. 逐条生成技术应答/偏差表：                                │
│      条款编号 | 条款原文 | 是否满足 | 偏离说明 |              │
│      响应摘要 | 证明材料名称 | 证明材料定位                   │
│      （页码/文件内位置，如"审计报告第3页"）|                 │
│      是否★必须项 | volume归属（资格|商务|技术）              │
│   2. 生成供货范围/设备清单表                                  │
│      每条含 volume 归属字段                                   │
│   3. 基于公司主数据 + 模板生成规则填充型内容：                │
│      - 投标函草稿、授权委托书草稿                             │
│      - 资格审查表                                             │
│      - 商务偏差表                                             │
│      每项含 volume 归属字段                                   │
│   4. 生成证据附件索引（对照条款→匹配已上传的证据文件）       │
│      每条含：条款编号、匹配的证据文件名、                     │
│      证据定位（文件内页码或段落位置）、匹配依据摘要           │
│                                                              │
│ 输出格式：结构化 JSON + Markdown 混合                        │
│ 输出 → s3/                                                   │
│   s3/technical_response.json   ← 技术应答表（含 volume 字段  │
│                                   和证据定位字段）            │
│   s3/deviation_table.json      ← 偏差表（含 volume 字段）    │
│   s3/equipment_list.json       ← 设备清单（含 volume 字段）  │
│   s3/form_fills.json           ← 规则填充内容（结构化，      │
│                                   每项含 volume、content、   │
│                                   template_type 字段）       │
│   s3/evidence_index.json       ← 条款↔证据匹配索引          │
│                                   （含证据定位信息）          │
│                                                              │
│ → 用户审核技术应答准确性，确认偏差项，补充遗漏               │
│   降级模式下此步骤主要审核规则填充内容                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐
│ 节点⑤  模型调用 [方案正文生成]  ← 可多模型并行              │
│                                                              │
│ 显示名称：「生成方案正文」                                    │
│ 输入：s1/* + s2/bid_blueprint.md + s3/*（如有）+ 资料库      │
│                                                              │
│ 执行前检查 s2/mode.json：                                    │
│   - normal 模式 → 按蓝图生成，引用 s3/ 中的技术应答数据     │
│   - degraded 模式 → 仅按蓝图中"待确认"章节生成建议性正文   │
│     不引用 s3/technical_response.json（降级模式下为空）       │
│     用户需逐章确认内容后才可继续                              │
│                                                              │
│ 提示词核心任务：                                              │
│   按蓝图中标注为"AI 生成"的章节，分组生成方案正文：         │
│                                                              │
│   组1: 供货与交付                                             │
│     供货方案、交货计划、运输保险                               │
│   组2: 实施与部署                                             │
│     项目实施方案、系统安装部署、集成调试方案                  │
│   组3: 运维与售后                                             │
│     售后服务方案、运维方案、故障响应、备品备件                │
│   组4: 培训与组织                                             │
│     培训方案、项目组织架构、团队成员安排                      │
│   组5: 质量与安全                                             │
│     质量保证措施、安全措施、保密措施                          │
│   组6: 工程建设（仅工程建设类项目）                          │
│     建设方案（各子系统）、施工方案、进度计划                  │
│   组7: 增值内容                                               │
│     重点难点分析、合理化建议、技术优势说明                    │
│                                                              │
│   具体生成哪些组由蓝图决定，蓝图里没要求的组跳过             │
│   每组内容基于资料库 RAG 检索历史标书和通用模板增强           │
│                                                              │
│ 输出 → s4/                                                   │
│   s4/group1_delivery.md                                      │
│   s4/group2_implementation.md                                │
│   s4/group3_service.md                                       │
│   s4/group4_training.md                                      │
│   s4/group5_quality.md                                       │

│   - ★必须项不满足预警：如存在★项实际不满足，标记为          │

│                                                              │
│ → 用户查看质检结果，决定是否回退修正                         │
│   （系统已支持回退到任意节点重新执行）                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐

│                                                              │

└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐

│                                                              │

│   export/卷一-资格文件.docx    ← 多卷模式                   │
│   export/卷二-商务文件.docx                                  │

│   export/export_checklist.md   ← 导出检查清单               │
│                                                              │
```

### 流程总览

```
输入转换 → 信息脱敏 → [分析招标] → [技术响应] → [方案生成] → [质检] → 信息恢复 → 文件导出
  ①          ②          ③            ④            ⑤          ⑥         ⑦          ⑧

节点类型：  输入转换  信息脱敏  模型调用×4                    信息恢复   文件导出
```

8 个节点，其中 4 个是模型调用节点（各自职责不同）。符合 IntelliFlow 的"同一类型可多次添加"设计。

## 八、可行性与限制（v2 修订）

### 可行

| 环节 | 可行性 | 说明 |
|------|--------|------|
| 招标文件分析 → 蓝图生成 | **高** | AI 擅长结构化分析，评分标准提取、条款分级标注均为成熟能力 |
| 技术应答/偏差表 | **高** | 逐条对照是 AI 最擅长的结构化任务，4 个项目样本都验证了统一模式 |
| 规则填充（投标函等） | **中高** | 基于主数据+模板填充，准确率高，但需人工确认措辞 |
| 方案正文（售后/培训/质量等通用方案） | **中高** | 有历史标书做 RAG 参考时质量好，无参考时偏泛 |
| 方案正文（建设方案/实施方案等专业方案） | **中** | 需要产品选型和项目现场信息，AI 可生成框架但细节需人工补充 |
| 合规质检 | **高** | 交叉校验、覆盖率检查是规则化任务 |
| 多卷宗导出 | **中** | 需要开发导出模板引擎，技术可行但工作量不小 |

### 限制与应对

| 限制 | 影响 | 应对 |
|------|------|------|
| 大文档解析质量 | 500+ 页的 docx 解析可能丢失表格结构和页码 | 系统层面需优化文档解析引擎，特别是表格提取能力 |
| 产品选型依赖 | 无产品信息时技术响应无法生成 | 节点③蓝图中标注"缺失：产品选型"，提示用户补充后再继续 |
| 图片/扫描件内容 | 200+ MB 的文件中大量内容是图片形式 | OCR 能力依赖系统输入转换节点，当前不追求完美解析 |
| 导出排版 | 不同招标方格式要求不同 | v1 提供通用 Word 模板，按需定制；后续支持多模板 |
| 证据材料装配 | 需人工上传真实证书扫描件 | 流程明确标注哪些是"证据挂载"，用户清楚需要准备什么 |

### AI 生成价值评估

不再按项目类型做先验判断。实际价值取决于单个项目的**内容复杂度**：

| 复杂度维度 | 高价值信号 | 低价值信号 |
|-----------|-----------|-----------|
| 方案正文量 | 需要建设方案、实施方案等数十页正文 | 主要是表格和承诺函 |
| 技术条款密度 | 50+ 条技术要求需逐条应答 | <10 条简单参数 |
| 证明材料密度 | 每条需附证明 | 只需声明"满足" |
| 评分维度数 | 评分表 10+ 维度 | 仅价格评分 |

## 九、与 v1 的主要变化总结

| 项 | v1 | v2 |
|----|----|----|
| 输入要求 | 招标文件必选 | 项目材料至少一种，招标文件优先 |
| 第一层内容分类 | 统一"AI 无法生成" | 拆分为规则填充/证据挂载/人工确认 |
| 模型调用节点数 | 3 个（分析+技术响应+方案，合并为"大节点"） | 4 个（分析、技术响应、方案生成、质检各独立） |
| 方案生成方式 | 一次性全部生成 | 按 7 个内容组分步生成 |
| 技术响应输出 | 纯 Markdown | 结构化 JSON + Markdown 混合 |
| 质检范围 | 完整性+一致性 | 合规检查+一致性检查+完整性检查 三层 |
| 导出 | 简单描述 | 支持单卷/多卷模式、封面、目录、附件装配 |
| 价值判断 | 按项目类型分高/低 | 按内容复杂度维度评估 |
| 证据管理 | 未涉及 | 条款→响应→证据引用闭环，质检验证完备性 |

## 十、v2.1 修订记录（基于评审反馈）

### 修复的问题

| 级别 | 问题 | 修复方式 |
|------|------|---------|
| **High** | "项目材料至少一种"未落地，节点③仍硬依赖 tender_doc.md | 节点③增加**降级模式分支**：无招标文件时生成建议性大纲，标注"待确认"，明确不支持技术应答/偏差表生成；文档类型描述同步更新 |
| **High** | 证据链不闭环，evidence_manifest.json 只有"文件名+分类"，后续节点无法真正匹配和校验 | evidence_manifest.json 扩展为**结构化元数据**：每条附件含解析文本摘要（前500字）、关键字段提取（证书编号、有效期、发证机构等）；质检节点对照元数据做匹配而非仅靠文件名猜测 |
| **High** | ★必须项质检定义为"是否全部回答满足"，会掩盖真实不满足 | 改为**三步合规检查**：①每个★项是否被明确响应（完整性）→ ②是否存在实际不满足项（真实性）→ ③不满足项标记为【投标无效风险】并**阻断导出**，要求用户显式确认处理 |
| **Medium** | 多卷宗导出缺少章节/附件归卷字段 | 节点③蓝图中每章增加 **volume 字段**（资格\|商务\|技术），导出节点按 volume 自动分拣归卷，不硬编码 |
| **Medium** | 公司主数据未物化为节点输入，重跑不稳定 | 节点①输出增加 **s1/company_snapshot.json**，从资料库固化公司简介、法人信息、资质清单、人员库摘要，确保可复现 |

### v2.2 补充修复

| 级别 | 问题 | 修复方式 |

### Open Questions 回应

| 问题 | 结论 |
|------|------|
| IntelliFlow 的输入转换是否会把证据附件自动解析成可检索文本？ | **是**。v4 需求文档明确规定"上传文件支持 Word/PDF/图片，系统自动解析为文本（PDF/Word 提取文字，图片 OCR）"。因此 evidence_manifest.json 可以包含解析摘要。但需注意：扫描件 OCR 质量可能不稳定，关键字段提取（证书编号、有效期等）需要额外的结构化抽取逻辑，建议在输入转换节点的系统处理中增加"证据附件字段提取"步骤 |
| 资料库引用在运行时是否会自动固化为本次流程的输入快照？ | **否，当前不会**。v4 需求中资料库是 RAG 索引，模型调用时通过检索获取相关段落，但不会自动生成固化快照。因此新增 s1/company_snapshot.json 是必要的——将公司主数据在输入转换阶段固化为本次流程的输入文件，确保节点④重跑时数据一致 |
